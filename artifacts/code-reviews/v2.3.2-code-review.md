## 2. Current Architecture and Request Flow

### Request Flow (all 4 endpoints)

```
HTTP Request
  → IssuesController (thin: validate path variables, delegate, return)
    → IssuesRequestMapper (DTO → orchestration model)
      → IssuesService (orchestrate: validate business rules, delegate to port, compose result)
        → IssuesPort (provider-agnostic interface)
          → GitLabIssuesAdapter (executeGitLabOperation wrapper → RestClient call → exception mapping)
            → GitLabIssuesMapper / GitLabIssueDetailMapper (GitLab DTO → orchestration model)
    → IssuesResponseMapper (orchestration model → response DTO)
  → HTTP Response (or GlobalExceptionHandler → ErrorResponse)
```

### Package Structure (40 production classes)

| Layer | Package | File Count | Responsibility |
|-------|---------|-----------|---------------|
| REST boundary | `orchestration/issues/rest/` | 1 controller, 2 mappers, 7 DTOs = 10 | HTTP entry, request/response mapping |
| Orchestration | `orchestration/issues/` | 1 service, 1 port, 7 models = 9 | Use-case coordination, business rules |
| Integration | `integration/gitlab/issues/` | 1 adapter, 2 mappers, 3 DTOs = 6 | GitLab-specific HTTP calls and mapping |
| Integration infra | `integration/gitlab/` | 4 (config, locator, URI factory, exception mapper) | Shared GitLab infrastructure |
| Common | `common/` | 4 (error types + exception handler + error response) | Cross-cutting error handling |
| Config | `config/` | 2 (properties records) | Externalized configuration |
| App root | root | 1 (`FlowOrchestratorApplication`) | Spring Boot entry point |

### Endpoints

| Method | Path | Controller Method | Response |
|--------|------|------------------|----------|
| `POST` | `/api/issues/search` | `getIssues()` | `SearchIssuesResponse` |
| `POST` | `/api/issues` | `createIssue()` | `IssueDto` (201) |

## 3. Strong Sides

1. **Clean ports-and-adapters boundary.** `IssuesPort` is a proper provider-agnostic interface. All 7 orchestration models carry zero GitLab-specific concerns. Constitution Principle 2 is well-respected.

2. **Explicit mapping at every layer boundary.** Request DTOs → orchestration models (via `IssuesRequestMapper`), orchestration models → response DTOs (via `IssuesResponseMapper`), GitLab DTOs → orchestration models (via `GitLabIssuesMapper`, `GitLabIssueDetailMapper`). No model leaks. Constitution Principle 8 is satisfied.

3. **Centralized exception handling.** `GlobalExceptionHandler` provides a single translation point. It correctly handles `ValidationException`, `IntegrationException`, `MethodArgumentNotValidException`, `HttpMessageNotReadableException`, `MethodArgumentTypeMismatchException`, and a catch-all. Constitution Principle 7 is satisfied.

4. **Immutable data models everywhere.** All models (orchestration, DTO, integration) are Java records with defensive copies of collection fields in compact constructors. Constitution Principle 6 is satisfied. The `Defensive Copy Gate` from code-guidance is met.

5. **Good test pyramid.** 92 tests across 4 levels:
   - Unit: mappers, service, DTOs, exception mapper, URI factory, project locator, adapter (with `MockRestServiceServer`)
   - Integration: `IssuesControllerIT` with `@WebMvcTest` covering all endpoints, validation, and error mapping
   - Component: `IssuesApiComponentTest` with WireMock for full HTTP flow
   - Karate: 7 `.feature` files for smoke testing against a running app

6. **Configuration is externalized, validated, and immutable.** `GitLabProperties` uses `@NotBlank` validation; `IssuesApiProperties` uses `@Min`/`@Max` with a cross-field invariant in the compact constructor. Constitution Principle 5 is satisfied.

7. **Structured logging with SLF4J.** All logging uses `@Slf4j` and parameterized messages. No secrets in log output. Context fields (count, resource, status, category) are consistently included at appropriate levels.

8. **`GitLabUriFactory` consolidates URL construction.** The v2.3.0 refactoring successfully eliminated repeated path building across adapter methods. Each adapter method now delegates to `gitLabUriFactory.projectResourcePath(RESOURCE_ISSUES)`.

9. **`executeGitLabOperation` pattern is clean.** The adapter wraps all RestClient calls in a generic try-catch that consistently maps `RestClientResponseException` and `RuntimeException` to typed `IntegrationException`. This eliminates exception-handling duplication.

10. **Code-guidance conventions are followed consistently.** `*Service`, `*Port`, `*Adapter`, `*Mapper`, `*Properties`, `*Request`, `*Response`, `*Dto`, `*Input` suffixes are all used as documented.

## 4. Findings and Gaps

### 4.1 Validation Pattern — `validateIssueId` and Annotation Alternatives

**What is happening now:**

[IssuesController.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/IssuesController.java#L82-L86) contains a private method:

```java
private void validateIssueId(final long issueId) {
    if (issueId <= 0) {
        throw new ValidationException(REQUEST_VALIDATION_FAILED, List.of(ISSUE_ID_POSITIVE_MESSAGE));
    }
}
```

This is called at the top of `deleteIssue()` and `getIssueDetail()` — every endpoint that takes `{issueId}` as a path variable. The pattern is consistent and works correctly. It was not introduced in the most recent run — it was properly reused from the existing pattern.

**Why it is a problem or risk:**

- **Scaling concern.** Every new controller (milestones, merge requests, label events) will need to repeat this pattern for its own entity IDs. This creates boilerplate across controllers.
- **Inconsistency with DTO validation.** `CreateIssueRequest` and `PaginationRequest` already use Jakarta Bean Validation annotations (`@NotBlank`, `@Positive`, `@Valid`). The `@Positive` annotation on `PaginationRequest.page` and `PaginationRequest.perPage` is functionally identical to `issueId > 0`. Having two different validation mechanisms for the same class of constraint breaks the single-validation-style principle.
- **Error message coupling.** The controller owns a custom error message string and throws `ValidationException` manually, while `@Positive` violations are caught by `MethodArgumentNotValidException` and translated via `GlobalExceptionHandler`. The user sees slightly different error structures for the same category of problem.

**What to prefer instead — Annotation-based approach:**

Spring MVC supports `@Positive` on `@PathVariable` parameters when the controller is annotated with `@Validated`:

```java
@Validated
@RestController
@RequestMapping("/api/issues")
public class IssuesController {

    @DeleteMapping("/{issueId}")
    public ResponseEntity<Void> deleteIssue(@PathVariable @Positive final long issueId) {
        // no manual validation needed
    }
}
```

When `issueId <= 0`, Spring throws `ConstraintViolationException`, which needs a handler in `GlobalExceptionHandler`.

**Trade-offs:**

| Aspect | Current (manual) | Annotation (`@Positive` + `@Validated`) |
|--------|-----------------|----------------------------------------|
| Boilerplate per endpoint | 1 method call | 0 |
| Boilerplate per controller | 1 private method + 2 constants | 0 |
| Consistency with DTO validation | Inconsistent (two mechanisms) | Consistent (one mechanism) |
| Error message control | Full control | Requires `@Positive(message = "...")` or handler customization |
| Required change | None | Add `@Validated` on class, `@Positive` on params, new `ConstraintViolationException` handler |
| Exception handler impact | None | Must add `ConstraintViolationException` handler to `GlobalExceptionHandler` to produce the same `ErrorResponse` shape |

**Recommendation:** Replace with `@Positive` + `@Validated`. The `ConstraintViolationException` handler is a one-time addition to `GlobalExceptionHandler` that pays off across all future controllers. This aligns with code-guidance: *"Validation happens early at the correct boundary"* — annotations are the idiomatic Spring way to validate at the REST boundary.

**Priority:** MEDIUM — not urgent, but should be addressed before the next capability is added to avoid repeating the manual pattern.

### 4.2 Adapter Naming and Role Clarity

**What is happening now:**

[GitLabIssuesAdapter.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/integration/gitlab/issues/GitLabIssuesAdapter.java) implements `IssuesPort` and has these responsibilities:

1. Build URIs using `GitLabUriFactory`
2. Execute HTTP calls via `RestClient`
3. Map exceptions via `GitLabExceptionMapper`
4. Delegate response mapping to `GitLabIssuesMapper` / `GitLabIssueDetailMapper`
5. Log integration-level context

After the v2.3.0 refactoring (URI factory extraction) and the mapper extraction, the adapter is now essentially a **thin coordination layer** between `RestClient`, `GitLabUriFactory`, `GitLabExceptionMapper`, and the mappers. It does not contain business logic. Each method follows the same pattern: call RestClient → map response → return orchestration model.

**Is it a thin wrapper?** Yes. Each method is 5-15 lines of actual logic (excluding logging), following a consistent template.

**Is "Adapter" the right name?**

Yes, and here is why:

- In the ports-and-adapters (hexagonal) architecture that this project follows, an **adapter** is exactly the component that implements a port by adapting an external system's protocol to the port's contract. `GitLabIssuesAdapter implements IssuesPort` is textbook hexagonal naming.
- The code-guidance document explicitly says: *"Use `*Adapter` for provider-facing implementations of ports."*
- Constitution Principle 2 says: *"Orchestration defines provider-agnostic ports (interfaces) and request/response contracts. Integration implements those ports."*
- "Thin" is not a problem — it is the goal. The adapter should be thin. Complex business logic in the adapter would be an architectural violation.

**Alternative names considered and rejected:**

| Name | Why not |
|------|---------|
| `GitLabIssuesClient` | Implies it is the HTTP client itself, which it is not — `RestClient` is the client. Also drops the ports-and-adapters semantics. |
| `GitLabIssuesGateway` | Valid in some architectures, but not used by the code-guidance conventions. Would be a deviation. |
| `GitLabIssuesService` | Conflicts with the `*Service` convention (reserved for orchestration). |

**Verdict:** `GitLabIssuesAdapter` is the correct name. No change recommended.

**One area to watch:** If the adapter grows beyond ~150 lines per method or starts doing business-level decisions (e.g., retry logic, result composition across multiple API calls), it should be split. Currently this is not a concern — the adapter is clean and focused.

### 4.3 Record Constructor Ergonomics

**What is happening now:**

All models across all layers are Java records. Record instantiation uses positional constructor arguments. Examples of the ergonomic cost:

**9-argument record** — [Issue.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/model/Issue.java):
```java
new Issue(101L, 42L, "Fix login", "desc", "opened", List.of("bug"), "john", "M1", 77L)
```

**10-argument record** — [IssueDetail.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/model/IssueDetail.java):
```java
new IssueDetail(42L, "Fix login bug", null, "opened", List.of(), List.of(), null,
    OffsetDateTime.parse("2026-01-04T15:31:51.081Z"),
    OffsetDateTime.parse("2026-03-12T09:00:00.000Z"), null)
```

**11-argument record** — [IssueDetailDto.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/dto/IssueDetailDto.java):
```java
new IssueDetailDto(42L, "Fix login bug", "SSO broken", "opened", List.of("bug"),
    List.of(new AssigneeDto(10L, "john.doe", "John Doe")),
    new MilestoneDto(5L, 3L, "Sprint 12", "active", "2026-04-30"),
    OffsetDateTime.parse("2026-01-04T15:31:51.081Z"),
    OffsetDateTime.parse("2026-03-12T09:00:00.000Z"), null, List.of())
```

**Why it is a problem:**

1. **Argument-order risk.** Swapping two `long` or two `String` arguments compiles cleanly but produces a silent bug. `Issue(id, issueId, ...)` has two consecutive `long` fields — swapping them would be invisible at compile time.
2. **Readability.** In mapper code and test fixtures, these calls are dense walls of positional values. A reader must count positions to understand which value maps to which field.
3. **Maintainability.** Adding a field to a record breaks every call site. With 9+ arguments, finding and updating all call sites is error-prone.

**Refactoring Options:**

| Option | Mechanism | Pros | Cons |
|--------|-----------|------|------|
| **A. Lombok `@Builder`** | Add `@Builder` to records | Named arguments via builder pattern; fluent; one field addition doesn't break existing sites if defaults are provided | Requires Lombok processing; slightly more bytecode; still need to remember required fields |
| **B. Static factory + builder (manual)** | Hand-write a builder inner class | No Lombok dependency; full control | Significant boilerplate; defeats the purpose of records being concise |
| **C. Lombok `@Builder` on records, production only** | Apply `@Builder` to orchestration models and DTOs with ≥6 fields; keep mappers using builders; tests use builders | Best balance — records stay immutable; construction in mappers/tests becomes readable; small records (2-4 fields) keep simple constructors | Needs Lombok on classpath (already present) |
| **D. Keep records, use static factory methods** | Add `Issue.of(...)` or `Issue.create(...)` with well-named params | Slightly better but still positional; doesn't solve the core problem for 9+ args |  |

**Recommendation: Option C — Lombok `@Builder` on records with ≥6 fields.**

The code-guidance already permits this: *"@Builder is used only when it improves readability for object construction"* and *"Prefer builder-style object construction over direct `new Class(...)` when it materially improves readability or reduces argument-order risk."*

Example after refactoring:

```java
@Builder
public record Issue(
        long id,
        long issueId,
        String title,
        @Nullable String description,
        String state,
        List<String> labels,
        @Nullable String assignee,
        @Nullable String milestone,
        @Nullable Long parent) {
    public Issue {
        labels = labels == null ? List.of() : List.copyOf(labels);
    }
}
```

Usage in mapper:
```java
return Issue.builder()
    .id(issueResponse.id())
    .issueId(issueResponse.iid())
    .title(issueResponse.title())
    .description(issueResponse.description())
    .state(issueResponse.state())
    .labels(issueResponse.labels())
    .assignee(assignee)
    .milestone(milestone)
    .parent(parent)
    .build();
```

**Records to apply `@Builder` to** (all have ≥6 fields):

| Record | Fields | Location |
|--------|--------|----------|
| `Issue` | 9 | `orchestration/issues/model/` |
| `IssueDetail` | 10 (compact constructor for defensive copy) | `orchestration/issues/model/` |
| `IssueDetail.MilestoneDetail` | 5 | inner record — borderline, optional |
| `IssueQuery` | 6 | `orchestration/issues/model/` |
| `IssueDto` | 9 | `orchestration/issues/rest/dto/` |
| `IssueDetailDto` | 11 | `orchestration/issues/rest/dto/` |
| `GitLabIssueResponse` | 8 | `integration/gitlab/issues/dto/` |
| `GitLabSingleIssueResponse` | 11 | `integration/gitlab/issues/dto/` |

**Important caveat:** Lombok `@Builder` on a record with a compact constructor requires care. Lombok generates the builder to call the canonical constructor, which invokes the compact constructor. The defensive copy logic in compact constructors is preserved. This has been verified with Lombok ≥1.18.30 and Java 21.

**Priority:** MEDIUM — improves readability and reduces argument-order risk. Best addressed before the next capability (label events, milestones) adds more models.

### 4.4 Error Code Semantics — `INTEGRATION_NOT_FOUND` for 404

**What is happening now:**

When GitLab returns HTTP 404, the [GitLabExceptionMapper](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/integration/gitlab/GitLabExceptionMapper.java#L19-L21) maps it to `ErrorCode.INTEGRATION_NOT_FOUND`. The [GlobalExceptionHandler](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/common/web/GlobalExceptionHandler.java#L82) maps this to HTTP 404.

The client sees:
```json
{
  "code": "INTEGRATION_NOT_FOUND",
  "message": "GitLab delete issue operation failed",
  "details": []
}
```

**Why it is a problem:**

`INTEGRATION_NOT_FOUND` semantically reads as "the integration itself was not found" (e.g., GitLab connectivity or project not found), rather than "the requested resource (issue) was not found through the integration." The user flagged this in v2.3.0 logs: *"I'd prefer 404, ISSUE_NOT_FOUND or ITEM_NOT FOUND, as integration not found refers more to gitlab connectivity issues rather than missing ticket."*

This is a valid concern. As the application grows to support milestones, merge requests, and label events, `INTEGRATION_NOT_FOUND` for all 404s becomes ambiguous — was the project not found, the issue not found, or the milestone not found?

**What to prefer instead:**

Add a resource-specific `RESOURCE_NOT_FOUND` error code (or even `ISSUE_NOT_FOUND` for issues), and make the exception mapper context-aware:

```java
// ErrorCode.java
RESOURCE_NOT_FOUND,  // replaces INTEGRATION_NOT_FOUND for 404

// Or finer-grained, if you want capability-specific error codes:
ISSUE_NOT_FOUND,
MILESTONE_NOT_FOUND,
```

The `GitLabExceptionMapper` already receives the `resource` parameter — it can use it to select the right error code.

**Recommendation:** Replace `INTEGRATION_NOT_FOUND` with `RESOURCE_NOT_FOUND` as the default 404 code. Keep the same HTTP 404 mapping in `GlobalExceptionHandler`. Optionally, allow adapters to pass capability-specific error codes for 404 contexts.

**Priority:** LOW — cosmetic but semantically cleaner. Can be addressed alongside the next capability.

### 4.5 Logging Consistency and Gaps

**Overall assessment: Logging is good but has specific gaps.**

**What is working well:**
- All controllers log request receipt and response return with business context (count, page, id, hasBody)
- `IssuesService` logs operation start and completion with relevant fields
- `GitLabIssuesAdapter` logs operation start, completion, warnings on HTTP failures, and errors on transport failures
- Mappers use `log.debug()` for field-level mapping details
- `GlobalExceptionHandler` logs all exception categories with appropriate levels (warn for validation/integration, error for unhandled)
- `GitLabProjectLocator` logs resolved project reference at startup

**Gaps identified:**

| # | Location | Gap | Impact |
|---|----------|-----|--------|
| G-1 | `IssuesController.deleteIssue()` | No log after successful deletion. `deleteIssue` logs entry but not completion — contrast with `getIssueDetail` which logs both. | Minor inconsistency in request lifecycle tracing |
| G-2 | `IssuesRequestMapper` | No logging at all. When request mapping produces unexpected results (e.g., null filters, default pagination), there is no trace. | Difficult to debug mapping issues in production without adding log after the fact |
| G-3 | `GitLabIssuesAdapter.getIssues()` | The `fetchIssues` method logs at `debug` level inside the URI builder lambda. This is unusual — logging inside a lambda passed to `uri()`. | Not harmful, but unconventional. If DEBUG is off, the lambda still allocates the log message |
| G-4 | `IssuesService.deleteIssue()` | Logs identically before and after: `"Deleting issue issueId={}"` and `"Issue deleted issueId={}"`. The post-delete log adds no new information since there is no return value. | Minor — the pre/post pattern is intentional for operation bracketing, but the post-log adds nothing diagnostic since there is no outcome to report |

**Recommendation:** G-1 (add delete completion log) is a 1-line fix. G-2 (mapper logging) is optional — mappers are pure functions and their behavior is covered by tests. G-3 and G-4 are informational, no action needed.

**Priority:** LOW — logging is already above average for this codebase size.

### 4.6 Mapper Duplication Between Layers

**What is happening now:**

The mapping from `AssigneeDetail` → `AssigneeDto` and `MilestoneDetail` → `MilestoneDto` appears in two places:

1. [GitLabIssueDetailMapper.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/integration/gitlab/issues/mapper/GitLabIssueDetailMapper.java#L18-L20) maps `GitLabAssigneeDetail` → `AssigneeDetail` (integration → orchestration)
2. [IssuesResponseMapper.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/mapper/IssuesResponseMapper.java#L37-L39) maps `AssigneeDetail` → `AssigneeDto` (orchestration → REST)

These are structurally similar: extracting the same 3 fields (`id`, `username`, `name`) from one record into another.

**Why it is not actually a problem:**

This is **correct per Constitution Principle 8** — mapping responsibility belongs to the layer that owns the transformation. The integration mapper maps GitLab shapes to orchestration models. The response mapper maps orchestration models to REST DTOs. Even though the transformations look similar, they cross different boundaries and must remain separate. Merging them would violate the layered architecture.

**Verdict:** No change recommended. This is working as designed.

### 4.8 IssuesService Validates Only perPage, Controller Validates Only issueId

**What is happening now:**

Validation is split across two locations:
- `IssuesController`: validates `issueId > 0` (path variable)
- `IssuesService`: validates `perPage <= maxPageSize` (business rule)

**Why this is an observation, not a gap:**

Both placements are architecturally correct. Path variable validation belongs at the REST boundary (Constitution Principle 1: *"Entry points in orchestration must stay thin: validate input…"*). Business rule validation (max page size depends on configuration) belongs in the service layer to stay transport-agnostic.

If `validateIssueId` moves to annotations (Finding 4.1), this split becomes even cleaner — all REST validation is annotation-driven, all business validation is in the service.

**Verdict:** No change needed. This is working as designed.

Since `CreateIssueRequest` has a compact constructor that copies labels, by the time the controller receives it, labels could still be null (compact constructor preserves null for "not provided"). This is correct but the null-check in the controller is a micro-smell — the controller shouldn't need to know about the DTO's nullability internals. | LOW |
| M-2 | `GitLabIssuesAdapter` | Multiple `@SuppressWarnings("null")` annotations on private methods. These suppress null-safety warnings from `RestClient.body()` calls. While harmless, they indicate the RestClient API's nullability annotations are fighting with the code. | LOW — cosmetic |
| M-3 | `IssuesControllerIT` | Uses deprecated `@MockBean` (deprecated in Spring Boot 3.4+). Should eventually migrate to `@MockitoBean`. | LOW — works fine currently but will produce warnings on upgrade |
| M-4 | `IssueFiltersRequest` | Override accessor methods (`labels()`, `assignee()`, `milestone()`) that wrap results in `List.copyOf()`. This means calling `.labels()` creates a new defensive copy every time. The compact constructor already copies, so the accessor copies are redundant — they copy an already-immutable list. | LOW — slight unnecessary allocation, no correctness issue |

## 6. Final Verdict

The `flow-orchestrator` codebase at v2.3.2 is **architecturally sound and well-organized**. Constitution principles are respected across all layers. The ports-and-adapters boundary is clean, mapping is explicit, and exception handling is centralized. The test pyramid is solid (92 tests across 4 levels), and configuration is properly externalized and validated.

The three user-challenged areas resolve as follows:

1. **`validateIssueId` → annotations:** Yes, replace with `@Positive` + `@Validated`. This is cleaner, more consistent with existing DTO validation, and eliminates per-controller boilerplate. **Priority 1.**
2. **Adapter naming:** `GitLabIssuesAdapter` is the correct name per hexagonal architecture and project conventions. It is a thin wrapper by design, and that is good. **No change needed.**
3. **Record constructor ergonomics:** Add Lombok `@Builder` to records with ≥6 fields. This is explicitly permitted by code-guidance and will significantly improve readability and safety. **Priority 2.**

The codebase is ready for the next capability (label events API). The recommended refactoring items (Priorities 1-2) should ideally be addressed before that capability is implemented to avoid propagating the manual validation pattern and positional constructors into new code.
