## 2. Strong Sides

### 2.1 Layer Boundaries Are Respected
Neither `GitLabIssueResponse` nor `GitLabCreateIssueRequest` appears in orchestration imports. Domain models carry no `@Json*` annotations. The constitution's Principle 2 (ports and adapters isolation) holds.

### 2.2 Service Layer Accepts Domain Models (v2.1.0 Fix Applied)
`GetIssuesService.getIssues(IssueQuery)` now takes the orchestration model instead of the API `IssuesRequest`. `CreateIssueService.createIssue(CreateIssueCommand)` follows the same pattern. Both services are reusable from non-HTTP entry points. The v2.1.0 code review finding (3.4) is resolved.

### 2.3 Immutable Records Throughout
All boundary models are Java records. Compact constructors apply `List.copyOf()` for defensive copies. No mutable state crosses layer boundaries.

### 2.4 Error Handling Has Structure
`GitLabExceptionMapper` centralizes HTTP status → `ErrorCode` translation. `GlobalExceptionHandler` catches all exception types and produces sanitized `ErrorResponse`. No stack traces leak to clients.

### 2.5 Lombok Usage Is Consistent Where Applied
All services and adapters use `@RequiredArgsConstructor` for constructor injection. All classes that log use `@Slf4j`. The v2.1.0 finding (3.3) about `IssuesRequestMapper` is resolved.

## 3. Architectural Findings

### 3.1 — Split Port/Adapter/Service Per Operation (Severity: High)

**Constitution reference**: Principle 13 — "Orchestration ports must be defined in terms of business intent, not provider mechanics. A new integration provider must be addable by creating a new adapter in the integration layer — with no changes to domain or orchestration code."

**Current state**: 2 endpoints produce 2 port interfaces, 2 services, and 2 adapters:

| Operation | Port | Service | Adapter |
|---|---|---|---|
| Search issues | `GetIssuesPort` | `GetIssuesService` | `GitLabIssuesAdapter` |
| Create issue | `CreateIssuePort` | `CreateIssueService` | `GitLabCreateIssueAdapter` |

**Problem**: The split is per-operation, not per-capability. If the "issues" capability grows (update, close, assign, transition), this pattern produces N ports, N services, and N adapters per operation. With 5 operations × 3 classes = 15 files for a single business capability.

More importantly, the two adapters share identical infrastructure: same `RestClient`, same `GitLabProjectLocator`, same `GitLabExceptionMapper`, same connection to the same GitLab project. The `GitLabCreateIssueAdapter` literally duplicates the exception handling pattern from `GitLabIssuesAdapter` (see 3.2).

**Recommendation**: One port per capability — `IssuesPort` with `getIssues()` and `createIssue()`. One adapter — `GitLabIssuesAdapter` implementing `IssuesPort`. One service — `IssuesService`. This aligns with Principle 13: the port expresses business capability, not HTTP verbs.

---

### 3.2 — Duplicate Exception Handling in Adapters (Severity: High)

**Current state**: Both adapters contain identical try/catch structures:

`GitLabIssuesAdapter.getIssues()`:
```java
catch (RestClientResponseException exception) {
    // map to IntegrationException via gitLabExceptionMapper
}
catch (RestClientException exception) {
    // map to IntegrationException via gitLabExceptionMapper
}
catch (RuntimeException exception) {
    // map to IntegrationException via gitLabExceptionMapper
}
```

`GitLabCreateIssueAdapter.createIssue()`:
```java
catch (RestClientResponseException exception) {
    throw handleHttpFailure(exception);
}
catch (RestClientException exception) {
    throw handleTransportFailure(exception.getClass().getSimpleName());
}
catch (RuntimeException exception) {
    throw handleTransportFailure(exception.getClass().getSimpleName());
}
```

**Problems**:
1. The `RestClientException` and `RuntimeException` catches perform identical logic. Since `RestClientException extends RuntimeException`, the `RestClientException` catch is redundant — a single `RuntimeException` catch would handle both.
2. The `GitLabCreateIssueAdapter` extracts `handleHttpFailure()` and `handleTransportFailure()` as private methods, while `GitLabIssuesAdapter` inlines the same logic. Two different styles for the same pattern.
3. Both adapters inject the same 4 dependencies: `RestClient`, `GitLabProjectLocator`, mapper, `GitLabExceptionMapper`.

**Recommendation**: Consolidate into a single adapter (see 3.1). Collapse the catch block to:
```java
catch (RestClientResponseException e) { throw mapHttpFailure(e); }
catch (RuntimeException e) { throw mapTransportFailure(e); }
```

### 3.3 — `CreateIssueService` Is a Pass-Through (Severity: Medium)

**Current state**: The entire `CreateIssueService`:
```java
public CreatedIssue createIssue(final CreateIssueCommand command) {
    log.info("Create issue request received ...");
    final CreatedIssue createdIssue = createIssuePort.createIssue(command);
    log.info("Issue created id={}", createdIssue.id());
    return createdIssue;
}
```

The service has no validation, no business logic, no orchestration — it logs and delegates. This is not a use-case service; it is a logging proxy. Constitution Principle 1 says entry points must "validate input, delegate to a use-case or service, map the result, return the response — nothing more." But the service itself should *contain* business behavior, not mirror the controller's delegation pattern.

Contrast with `GetIssuesService` which performs `validatePerPage()` — that is genuine orchestration logic.

**Recommendation**: When the port is consolidated into `IssuesPort`, the single `IssuesService` will contain both `getIssues()` (with validation) and `createIssue()`. The create path currently has no business logic, but the service exists as the place to add it (e.g., title length limits, label validation, duplicate detection). The pass-through is acceptable only if the architecture anticipates growth. Document this intent.

### 3.4 — `@ConditionalOnBean` on `CreateIssueService` (Severity: Medium)

```java
@Service
@ConditionalOnBean(CreateIssuePort.class)
@RequiredArgsConstructor
public class CreateIssueService {
```

This annotation means `CreateIssueService` is only registered if a `CreateIssuePort` bean exists. This is unusual for a port-and-adapter architecture where the adapter *always* exists in the application context.

**Problem**: If someone removes `@Component` from `GitLabCreateIssueAdapter` by accident, the service silently disappears. There is no startup failure — the controller's `@RequiredArgsConstructor` will fail instead with a confusing "No qualifying bean of type CreateIssueService" error rather than the clear "No qualifying bean of type CreateIssuePort" that constructor injection would produce.

**Recommendation**: Remove `@ConditionalOnBean`. If the port is missing, the application should fail-fast at startup with a clear error (which constructor injection already provides). `@ConditionalOnBean` is a legitimate pattern for optional integrations, but issue creation is not optional — it is a core feature.

### 3.5 — Controller Has 6 Injected Dependencies (Severity: Medium)

```java
private final GetIssuesService getIssuesService;
private final IssuesRequestMapper issuesRequestMapper;
private final IssuesResponseMapper issuesResponseMapper;
private final CreateIssueService createIssueService;
private final CreateIssueRequestMapper createIssueRequestMapper;
private final CreateIssueResponseMapper createIssueResponseMapper;
```

Six constructor parameters for a 2-endpoint controller. The root cause: per-operation mappers × per-direction mappers = 4 mapper beans + 2 services.

**Impact**: Every new operation adds 3 dependencies (service + request mapper + response mapper). At 5 operations, the controller has 15 injected dependencies.

**Recommendation**: After consolidating services (3.1) and mappers (see 5.2), the controller should have: 1 service + 1 request mapper + 1 response mapper = 3 dependencies.

---

### 3.6 — `GitLabExceptionMapper` Message Is Hardcoded for Retrieval (Severity: Low)

```java
final String failureMessage = "Unable to retrieve " + resource + " from GitLab";
```

The message always says "retrieve" even for `createIssue` operations. The `GitLabCreateIssueAdapter` passes `resource = "create-issue"`, so the resulting message is: *"Unable to retrieve create-issue from GitLab"*.

**Recommendation**: Change to a neutral message: `"GitLab " + resource + " operation failed"` or accept a verb parameter.

## 4. Package Structure Analysis

### Current Layout

```
com.gitlabflow.floworchestrator/
├── common/
│   ├── errors/          IntegrationException, ErrorCode, ValidationException
│   └── web/             GlobalExceptionHandler, ErrorResponse
├── config/              GitLabProperties, IssuesApiProperties
├── domain/              (empty — only package-info.java)
├── integration/
│   └── gitlab/
│       ├── GitLabRestClientConfig, GitLabExceptionMapper, GitLabProjectLocator
│       └── issues/
│           ├── GitLabIssueResponse (+3 nested records)
│           ├── GitLabIssueMapper, GitLabCreateIssueMapper
│           ├── GitLabCreateIssueRequest
│           ├── GitLabIssuesAdapter, GitLabCreateIssueAdapter
│
└── orchestration/
    └── issues/
        ├── Issue, IssuePage, IssueQuery, IssueState, CreatedIssue, CreateIssueCommand
        ├── GetIssuesPort, CreateIssuePort
        ├── GetIssuesService, CreateIssueService
        └── api/
            ├── IssuesController
            ├── IssuesRequest, IssuesResponse, PaginationRequest, IssueFiltersRequest
            ├── IssueResponseItem, CreateIssueRequest, CreateIssueResponse
            ├── IssuesRequestMapper, IssuesResponseMapper
            └── CreateIssueRequestMapper, CreateIssueResponseMapper
```

### Problems

**P1 — `domain/` is empty.** The `package-info.java` says "Core domain contracts and business concepts," but all business models (`Issue`, `IssuePage`, `IssueQuery`, `IssueState`) live in `orchestration.issues`. This contradicts the v2.1.0 review's architecture table, which described `domain.issues` as holding "Pure business models." Either move models to `domain/` or remove the empty `domain/` package.

**Decision**: The constitution's Principle 3 requires domain classes to have zero framework dependencies. The current models (`Issue`, `IssuePage`, `IssueQuery`, `IssueState`) already satisfy this — they are pure records. However, they also serve as port contract types (used in `GetIssuesPort.getIssues(IssueQuery): IssuePage`). In this project's current scale, orchestration models *are* the domain models. Moving them to `domain/` adds a package without adding a boundary.

**Recommendation**: Remove the `domain/` package (it is a forward placeholder that adds confusion). Keep models in `orchestration/issues/`. If domain logic (business rules, aggregates) is introduced later, create `domain/` at that time with a real reason.

**P2 — `orchestration/issues/` is flat with 12 files.** Ports, models, and services share the same package. At current size this is tolerable but becomes unnavigable as operations grow.

**Recommendation**: Introduce sub-packages only when the flat package exceeds ~15 files. The current 12 files (6 models + 2 ports + 2 services) are at the edge. After consolidation (3.1), this drops to 6 models + 1 port + 1 service = 8 files — comfortably flat.

**P3 — `orchestration/issues/api/` holds 11 files.** 7 DTOs + 4 mappers + 1 controller. After consolidation, this drops to 7 DTOs + 2 mappers + 1 controller = 10 files. Still dense. Consider `api/request/`, `api/response/`, `api/mapper/` sub-packages only if this grows further.

**P4 — `integration/gitlab/issues/` mixes models, mappers, and adapters.** 7 files (2 adapters, 2 mappers, 2 model records, 1 response with 3 nested records). After consolidation: 1 adapter, 1 mapper, 2 model records = 4 files. Flat package works.

### Proposed Layout (Post-Consolidation)

```
com.gitlabflow.floworchestrator/
├── common/
│   ├── errors/          IntegrationException, ErrorCode, ValidationException
│   └── web/             GlobalExceptionHandler, ErrorResponse
├── config/              GitLabProperties, IssuesApiProperties
├── integration/
│   └── gitlab/
│       ├── GitLabRestClientConfig, GitLabExceptionMapper, GitLabProjectLocator
│       └── issues/
│           ├── GitLabIssueResponse (+nested)
│           ├── GitLabCreateIssueRequest
│           ├── GitLabIssuesMapper              ← merged mapper
│           └── GitLabIssuesAdapter             ← merged adapter (impl IssuesPort)
│
└── orchestration/
    └── issues/
        ├── Issue, IssuePage, IssueQuery, IssueState
        ├── CreatedIssue, CreateIssueInput      ← renamed from CreateIssueCommand
        ├── IssuesPort                          ← merged port
        ├── IssuesService                       ← merged service
        └── api/
            ├── IssuesController
            ├── IssuesRequest, IssuesResponse, PaginationRequest, IssueFiltersRequest
            ├── IssueResponseItem, CreateIssueRequest, CreateIssueResponse
            ├── IssuesRequestMapper             ← absorbs CreateIssueRequestMapper
            └── IssuesResponseMapper            ← absorbs CreateIssueResponseMapper
```

**Delta**: 40 files → 31 files (−9). Controller dependencies: 6 → 3.

## 5. Model Proliferation Analysis

### 5.1 — Model Inventory for "Issues" Capability

The project has **2 API endpoints** and **18 model classes** (including nested records):

| Layer | Models | Count |
|---|---|---|
| Orchestration | `Issue`, `IssuePage`, `IssueQuery`, `IssueState`, `CreatedIssue`, `CreateIssueCommand` | 6 |
| API | `IssuesRequest`, `PaginationRequest`, `IssueFiltersRequest`, `IssuesResponse`, `IssueResponseItem`, `CreateIssueRequest`, `CreateIssueResponse` | 7 |
| Integration | `GitLabIssueResponse`, `GitLabAssignee`, `GitLabMilestone`, `GitLabEpic`, `GitLabCreateIssueRequest` | 5 |
| **Total** | | **18** |

**Ratio**: 9 models per endpoint. For a clean-architecture project, the expected overhead is 4-5 models per endpoint (1 API request, 1 API response, 1 orchestration model, 1 integration response, and optionally 1 integration request for writes).

### 5.2 — Mirror Models (1:1 Duplicates)

| API model | Orchestration model | Fields identical? |
|---|---|---|
| `IssueResponseItem` | `Issue` | **Yes** — same 8 fields, same types, same names |
| `CreateIssueResponse` | `CreatedIssue` | **Yes** — same 4 fields (`id`, `title`, `description`, `labels`) |

**Constitution reference**: Principle 8 requires "explicit mapping at every layer boundary." The existence of separate API and orchestration models is architecturally correct. However, the current `IssuesResponseMapper.toItem()` is a pure field-copy:

```java
private IssueResponseItem toItem(final Issue issue) {
    return new IssueResponseItem(
            issue.id(), issue.title(), issue.description(),
            issue.state(), issue.labels(), issue.assignee(),
            issue.milestone(), issue.parent());
}
```

And `CreateIssueResponseMapper.toResponse()` is identical:
```java
return new CreateIssueResponse(
        createdIssue.id(), createdIssue.title(),
        createdIssue.description(), createdIssue.labels());
```

**Verdict**: The separation is correct per the constitution. API records may diverge (adding `_links`, renamed fields for API consumers, hiding internal fields). Do not merge them. But do not create *more* mirror models in future features unless there is a presentation divergence reason. Document this policy in code-guidance.

### 5.3 — `CreateIssueCommand` Naming

**Current**: `CreateIssueCommand` in `orchestration.issues`.

**Problem**: "Command" implies a CQRS pattern (command bus, command handler, command metadata). This project has no command bus, no event sourcing, no CQRS separation. The record is a simple data carrier — `title`, `description`, `labels`. The `Command` suffix misleads readers into expecting enterprise dispatch infrastructure.

Constitution Principle 14: "Role-based naming suffixes must reflect the actual responsibility of the component. Use them only when the responsibility is genuine."

**Recommendation**: Rename to `CreateIssueInput`. Neutral, descriptive, no pattern implication.

### 5.4 — `IssueFiltersRequest` + `PaginationRequest` Nesting Depth

The search request has 3 levels of nesting: `IssuesRequest` → `PaginationRequest` + `IssueFiltersRequest`. This means the mapper must navigate: `request.pagination().page()`, `request.filters().labels()`.

The `IssuesRequestMapper` spends 25 lines extracting values through `Optional.ofNullable()` chains to handle all the nullability combinations:
```java
final int page = ofNullable(pagination).map(PaginationRequest::page).orElse(DEFAULT_PAGE);
final int perPage = ofNullable(pagination).map(PaginationRequest::perPage).orElse(issuesApiProperties.defaultPageSize());
final IssueState state = ofNullable(filters).map(IssueFiltersRequest::state).map(IssueState::fromValue).orElse(null);
```

**Verdict**: The nesting is debatable. It provides clean JSON grouping (`{ "pagination": { ... }, "filters": { ... } }`), but the cost is mapper complexity. This is a style choice, not a defect. Keep it, but add `@Nullable` annotations to `IssuesRequest` components to make nullability explicit at the type level.

### 5.5 — Defensive Copy Inconsistency Across Records

Six records have defensive copy constructors. Patterns vary:

| Record | Pattern |
|---|---|
| `CreateIssueCommand` | `labels = labels == null ? List.of() : List.copyOf(labels)` |
| `CreatedIssue` | `labels = labels == null ? List.of() : List.copyOf(labels)` |
| `CreateIssueResponse` | `labels = labels == null ? List.of() : List.copyOf(labels)` |
| `CreateIssueRequest` | `labels = labels == null ? null : labels.stream().filter(Objects::nonNull).toList()` |
| `IssueFiltersRequest` | Uses `sanitize()` method with stream filtering |
| `ErrorResponse` | `details = details == null ? List.of() : List.copyOf(details)` |

**Problems**:
- `CreateIssueRequest` preserves null (coerces to `null`, not `List.of()`). The other records coerce null to empty list. The semantics diverge without documentation.
- `CreateIssueRequest` also filters out null elements from the list — the only record doing so. This mixes validation with construction.
- `IssueFiltersRequest` uses a named `sanitize()` utility method while others inline the transformation.

**Recommendation**: Standardize on one pattern in code-guidance:
- API input records: preserve `null` (means "not provided"), filter out null elements → `labels = labels == null ? null : labels.stream().filter(Objects::nonNull).toList()`
- Orchestration/response records: coerce `null` to empty list → `labels = labels == null ? List.of() : List.copyOf(labels)`

Document this distinction. It is intentional: API records need to distinguish "not provided" from "empty list" for optional fields.

## 6. Naming Consistency Audit

### 6.1 — Port Naming: Verb-Prefixed vs Capability-Based

| Current | Pattern |
|---|---|
| `GetIssuesPort` | verb + plural noun |
| `CreateIssuePort` | verb + singular noun |

**Problems**:
- Inconsistent plurality: `Issues` vs `Issue`.
- Verb-prefix splits the capability into N interfaces, one per operation.
- Constitution Principle 13: "Orchestration ports must be defined in terms of business intent, not provider mechanics." The business intent is "issues management," not "getting issues" and "creating an issue."

**Recommended**: `IssuesPort` with methods `getIssues(IssueQuery)` and `createIssue(CreateIssueInput)`.

### 6.2 — Service Naming: Verb-Prefixed vs Capability-Based

| Current | Pattern |
|---|---|
| `GetIssuesService` | verb + plural noun |
| `CreateIssueService` | verb + singular noun |

Same issue as 6.1. **Recommended**: `IssuesService`.

### 6.3 — Adapter Naming: Inconsistent Compound Names

| Current | Implements |
|---|---|
| `GitLabIssuesAdapter` | `GetIssuesPort` |
| `GitLabCreateIssueAdapter` | `CreateIssuePort` |

The first adapter is named after the capability (`Issues`). The second is named after the operation (`CreateIssue`). Different naming strategies for the same layer.

**Recommended**: `GitLabIssuesAdapter` (single adapter implementing `IssuesPort`).

### 6.4 — Mapper Naming: 4 Different Styles

| Mapper | Location | Pattern |
|---|---|---|
| `IssuesRequestMapper` | orchestration/api | capability + direction |
| `IssuesResponseMapper` | orchestration/api | capability + direction |
| `CreateIssueRequestMapper` | orchestration/api | **operation** + direction |
| `CreateIssueResponseMapper` | orchestration/api | **operation** + direction |
| `GitLabIssueMapper` | integration/gitlab | provider + **singular** noun |
| `GitLabCreateIssueMapper` | integration/gitlab | provider + **operation** |

The search-related mappers use the capability name (`Issues`). The create-related mappers use the operation name (`CreateIssue`). The integration mappers split inconsistently between a capability mapper and an operation mapper.

**Recommended**:
- API layer: `IssuesRequestMapper` (absorb `CreateIssueRequestMapper.toCommand()` as `toCreateInput()`), `IssuesResponseMapper` (absorb `CreateIssueResponseMapper.toResponse()` as `toCreateResponse()`)
- Integration layer: `GitLabIssuesMapper` (absorb `GitLabCreateIssueMapper`)

### 6.5 — Orchestration Model Naming: Mixed Tense and Semantics

| Model | Pattern | Issue |
|---|---|---|
| `Issue` | noun | Clean |
| `IssuePage` | noun + container | Clean |
| `IssueQuery` | noun + intent | Clean |
| `IssueState` | noun + qualifier | Clean |
| `CreatedIssue` | **past tense** + noun | Past-tense prefix implies event, not data carrier |
| `CreateIssueCommand` | **verb** + noun + **CQRS suffix** | Implies enterprise dispatch pattern |

**Problem**: `Issue` is the read model, `CreateIssueCommand` is the write input, `CreatedIssue` is the write output. The naming uses three different semantic styles: bare noun, imperative verb + pattern suffix, past participle + noun.

**Recommended**:
- `CreateIssueCommand` → `CreateIssueInput` (neutral data carrier name)
- `CreatedIssue` → `CreatedIssue` (acceptable — describes what the object represents: an issue that was created, with a subset of `Issue` fields; could also be `CreateIssueResult` for symmetry)

### 6.6 — String Constants Duplication Across Adapters

| Constant | `GitLabIssuesAdapter` | `GitLabCreateIssueAdapter` |
|---|---|---|
| `SOURCE_GITLAB` | `"gitlab"` | `"gitlab"` |
| `RESOURCE` | `"project issues"` (local var) | `"create-issue"` (constant) |
| Log message template | inline string | `LOG_REQUEST_FAILURE`, `LOG_TRANSPORT_FAILURE` constants |

The same `"gitlab"` string is defined in both adapters. One adapter uses string constants for log messages; the other inlines them. Different approaches for identical concerns.

**Recommendation**: After port consolidation, this centralizes naturally. If keeping separate adapters, extract `SOURCE_GITLAB` to a shared constant in `integration.gitlab` package.
