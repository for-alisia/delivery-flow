## 2. Current architecture and request flow

### Current structure

The `orchestration` package is currently carrying the right responsibilities for this feature:

- capability-root services and provider-agnostic ports
- orchestration models that are independent of GitLab transport details
- thin REST entry points under `rest`
- REST DTOs and mappers that adapt HTTP payloads to orchestration contracts
- request validators for rules that are not cleanly expressible with bean validation alone

For milestones, the package split is coherent:

- `orchestration/milestones/MilestonesService.java` owns the use-case orchestration
- `orchestration/milestones/MilestonesPort.java` defines the integration contract
- `orchestration/milestones/model/*` owns the provider-neutral search input and shared milestone model
- `orchestration/milestones/rest/*` owns HTTP validation, request mapping, and response mapping

That structure matches `constitution.md` and `code-guidance.md`: the controller stays in orchestration, GitLab specifics stay in integration, and the shared milestone concept is now reusable by both the milestone API and issue detail responses.

### Request flow

Current request flow for `POST /api/milestones/search`:

1. `MilestonesController.searchMilestones()` accepts an optional body.
2. `MilestonesRequestMapper` converts the request into `SearchMilestonesInput`, applying default `ACTIVE` state and trimming `titleSearch`.
3. `MilestonesRequestValidator` validates `milestoneIds` rules that are intentionally deferred until after mapping.
4. `MilestonesService` logs the normalized search context and delegates to `MilestonesPort`.
5. `GitLabMilestonesAdapter` builds the provider request and delegates pagination traversal to `GitLabOffsetPaginationLoader`.
6. `GitLabOperationExecutor` centralizes GitLab HTTP/transport failure mapping and safe logging.
7. `GitLabMilestonesMapper` maps provider DTOs into the shared orchestration `Milestone` model.
8. `MilestonesResponseMapper` maps the orchestration model back to `SearchMilestonesResponse`.

### What is working well

- The service/port/adapter split is clean and easy to extend.
- Shared milestone extraction was done in the right place. Issue detail mapping now reuses the same model and DTO rather than inventing a second shape.
- Pagination was extracted into shared GitLab infrastructure instead of being reimplemented inside the milestone adapter.
- Tests prove the feature across unit, integration, component, and Karate smoke levels.

### What would be cleaner if refactored

The next cleanup should not reorganize the overall milestone package. The structure is already sound. The cleaner refactor is smaller and more surgical:

- standardize controller logging and controller method shape
- remove repeated null-suppression workarounds from GitLab adapters
- standardize the null-preserving immutable-list copy pattern used at HTTP/orchestration boundaries
- centralize small cross-cutting helpers only where repetition is now clearly spreading

## 3. Strong sides

### 3.1 Layering and separation of concerns are solid

The implementation follows the expected thin-controller, service, port, adapter, mapper flow. No GitLab DTOs or GitLab query mechanics leak into orchestration models. This is a clear pass against the constitution’s boundary rules.

### 3.2 Shared milestone reuse was implemented in the right direction

`Milestone` and `MilestoneDto` were extended once and reused from both the milestone endpoint and issue detail mapping. The additive fields (`description`, `startDate`) are proved in both `IssuesResponseMapperTest` and `GitLabIssueDetailMapperTest`, which is the right reuse discipline.

### 3.3 Logging is mostly good below the REST boundary

Logging is strongest in the service, adapter, pagination loader, and shared GitLab executor:

- `MilestonesService` logs normalized search context and completion summary
- `GitLabMilestonesAdapter` logs intent and result summary
- `GitLabOffsetPaginationLoader` logs aggregated page count, item count, and duration
- `GitLabOperationExecutor` consistently logs mapped GitLab failures without leaking sensitive data

That gives good supportability for successful flows and downstream failures.
