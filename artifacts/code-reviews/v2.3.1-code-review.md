## 2. Current Smoke-Test Architecture

### What exists today

The smoke-test mechanism is a 100-line bash script ([scripts/smoke-test.sh](../../scripts/smoke-test.sh)) that:

1. Requires a **running application** (`BASE_URL` defaults to `http://localhost:8080`)
2. Uses `curl` to call each endpoint with hardcoded payloads
3. Checks HTTP status codes and field presence via `grep`
4. Implements a **create → extract-id → delete → delete-again** flow for the delete lifecycle
5. Reports pass/fail counts to stdout

### Current test scenarios covered by smoke-test.sh

| # | Scenario | Method | Validation |
|---|----------|--------|------------|
| 1 | Health check | `GET /actuator/health` | HTTP 200 |
| 2 | Default search (empty body) | `POST /api/issues/search` | HTTP 200 |
| 3 | Filtered search | `POST /api/issues/search` | HTTP 200 |
| 4 | Create issue | `POST /api/issues` | HTTP 201 |
| 5 | Create validation error (blank title) | `POST /api/issues` | HTTP 400 |
| 6 | Create for delete flow | `POST /api/issues` | Extract `issueId` |
| 7 | Delete existing issue | `DELETE /api/issues/{id}` | HTTP 204 |
| 8 | Delete already-deleted issue | `DELETE /api/issues/{id}` | HTTP 404 |
| 9 | Delete validation error (id=0) | `DELETE /api/issues/0` | HTTP 400 |
| 10 | `issueId` field presence in search | `POST /api/issues/search` | `grep "issueId"` |
| 11 | `issueId` field presence in create | `POST /api/issues` | `grep "issueId"` |

### Strengths of current approach

- Covers all three API operations plus validation errors
- Tests the full create → delete lifecycle with real GitLab API (via `application-local.yml` PAT)
- Fast to run, zero dependencies beyond `curl`
- The `extract_issue_id` + chained delete flow is a genuine end-to-end scenario

### Weaknesses of current approach

| Weakness | Impact |
|----------|--------|
| **No response body validation** — only checks HTTP status and `grep` for field names | Cannot catch field value regressions, wrong data types, or missing fields beyond `issueId` |
| **Fragile id extraction** — `sed` regex to parse JSON is brittle | Could silently break on response format changes |
| **No cleanup guarantee** — if the script fails mid-run, created issues are left on GitLab | Pollutes the test project over time |
| **Not integrated into Maven lifecycle** — runs as a separate bash script | Cannot be gated in CI, not part of `mvn verify`, no report artifacts |
| **No parallel execution** — all checks run sequentially | Not a problem today (11 checks), but won't scale |
| **No structured reporting** — stdout text only | No CI-consumable reports (JUnit XML, HTML) |
| **Duplicated request payloads** — same payloads appear in `smoke-test.sh`, `issues.http`, and component tests | Maintenance drift risk |

## 3. Karate Framework Fitness Assessment

### What is Karate

Karate (v2.0.0, released March 2026) is an open-source Java-based API testing framework that:

- Uses a **Gherkin-like DSL** (`.feature` files) for readable, no-code API test definitions
- Provides **built-in JSON assertion** with fuzzy matchers (`#string`, `#number`, `#notnull`, schema validation)
- Runs via **JUnit 5** (`Runner.path(...).parallel(n)`) — integrates into Maven lifecycle
- Supports **environment-specific config** via `karate-config.js` and `karate.env`
- Generates **HTML reports** and **JUnit XML** out of the box
- Requires **Java 21+** (virtual threads) — matches project requirement
- Single dependency: `io.karatelabs:karate-core:2.0.0` (test scope)
- Supports **chained scenarios** (create → use response → delete) natively via variable passing
- Supports **tags** (`@smoke`, `@regression`) for selective execution
- MIT licensed, 8.8k GitHub stars, actively maintained

### Fit assessment against project requirements

| Requirement | Karate Fit | Notes |
|-------------|------------|-------|
| Java 21 / Spring Boot 3.3.5 compatibility | **Yes** | Karate v2.0.0 requires Java 21+, aligns perfectly |
| Maven integration | **Yes** | Single test-scope dependency, runs via `mvn test` or failsafe |
| Tests against running application (real GitLab) | **Yes** | `karate-config.js` can read `baseUrl` from env; feature files call real endpoints |
| Response body validation | **Yes** | `match` keyword with deep JSON assertions, fuzzy matchers, schema validation |
| Chained scenarios (create → fetch → delete) | **Yes** | Variables persist within a `Scenario` or across `Scenario Outline`; `def issueId = response.issueId` |
| Cleanup / teardown | **Yes** | `karate.configure('afterScenario', ...)` or explicit delete steps |
| CI-ready reporting | **Yes** | HTML + JUnit XML reports generated automatically |
| Environment switching (local vs CI) | **Yes** | `karate.env` + `karate-config.js` for base URL and auth |
| Existing test pyramid compatibility | **Yes** | Karate tests sit alongside or replace component tests; won't conflict with unit/integration |
| Parallel execution | **Yes** | `Runner.path(...).parallel(n)` |
| Constitution compliance (no secrets in code) | **Yes** | `karate-config.js` can read env vars or Java system properties for tokens |

### Concerns and mitigations

| Concern | Mitigation |
|---------|------------|
| **New dependency / learning curve** | Karate DSL is minimal; feature files are self-documenting. The project already has 3 test tool dependencies (JUnit 5, WireMock, Mockito). |
| **Karate v2.0.0 is very new** (March 2026) | v2.0.0 is GA, not RC. The project already targets Java 21 which Karate v2 requires. Fallback: pin to `1.5.2` if v2 issues arise. |
| **Feature files are not Java** | This is intentional — smoke/E2E tests should be readable by non-developers. The test-instructions allow component tests in `src/test/component/java`; Karate fits as a peer alongside or replacing the bash smoke tests. |
| **JaCoCo coverage** | Karate tests run via failsafe (not surefire), so they don't inflate unit coverage. This is correct behavior — smoke tests should not count toward the 85% line coverage gate. |

## 4. Comparison: curl vs Karate

| Dimension | curl (current) | Karate (proposed) |
|-----------|---------------|-------------------|
| Response validation | Status code + grep | Deep JSON match + schema |
| Variable passing (create → delete) | `sed` regex extraction | `def id = response.issueId` |
| Cleanup on failure | None | `afterScenario` hooks or `karate.call` |
| Maven integration | External script | `mvn test` / `mvn verify` |
| Reporting | stdout text | HTML + JUnit XML |
| Environment config | `$1` argument | `karate.env` + `karate-config.js` |
| Maintenance cost | Bash + curl expertise | Feature file DSL (minimal) |
| CI gating | Manual | Automatic via Maven |
| Readability | Moderate (bash) | High (Gherkin-like) |
| Parallel execution | None | Built-in |

**Verdict: Karate is a strong replacement.** It addresses every weakness of the curl approach while preserving the strengths (real HTTP calls against a running app with real GitLab).

## 5. Recommendation and Implementation Plan

### Overall recommendation

**Proceed with Karate adoption.** It is a natural fit for this project's tech stack (Java 21, Maven, Spring Boot), test philosophy (real HTTP verification), and quality goals (CI-ready gating).

### Placement in test pyramid

Per [test-instructions.instructions.md](../../.github/instructions/test-instructions.instructions.md), the project has three test levels:

- `src/test/java` — unit tests (`*Test`)
- `src/test/integration/java` — Spring slice tests (`*IT`)
- `src/test/component/java` — running-app tests (`*ComponentTest`)

Karate API tests are **running-application tests** that hit real HTTP endpoints (local app + real GitLab). They belong in the **component** test tier or a new **smoke/e2e** tier.

**Recommended placement:**

```
src/test/karate/
├── karate-config.js          # environment config (baseUrl, PAT from env var)
└── issues/
    ├── IssuesKarateTest.java  # JUnit 5 runner
    ├── issues-search.feature  # search scenarios
    ├── issues-create.feature  # create scenarios
    ├── issues-delete.feature  # delete scenarios
    └── issues-lifecycle.feature  # create → fetch → delete chain
```

**Rationale for `src/test/karate/` vs `src/test/component/`:**
- Karate tests require a pre-started application and a real GitLab project — they are closer to smoke/E2E than component tests
- The existing component tests use WireMock stubs — mixing Karate (real HTTP) and WireMock (stubbed HTTP) in the same source root could cause confusion
- A dedicated source root makes it clear these are opt-in, environment-dependent tests
- `build-helper-maven-plugin` already handles multiple test source roots

### Maven integration approach

```xml
<!-- New dependency in pom.xml -->
<dependency>
    <groupId>io.karatelabs</groupId>
    <artifactId>karate-core</artifactId>
    <version>2.0.0</version>
    <scope>test</scope>
</dependency>
```

**Execution strategy — fully isolated from local quick checks:**

The key design constraint: Karate tests must **never** run during `mvn test`, `mvn verify`, `scripts/verify-quick.sh`, or `scripts/final-check.sh`. They only execute on explicit opt-in.

- Add `src/test/karate/java` as a test source via `build-helper-maven-plugin`
- Name the runner class `*KarateTest.java`
- **Excluded from surefire** (`mvn test`): surefire already excludes `*IT.java` and `*ComponentTest.java`; add `**/*KarateTest.java` to the exclude list for safety
- **Excluded from default failsafe** (`mvn verify`): the existing failsafe config includes only `*IT.java` and `*ComponentTest.java` — `*KarateTest.java` is not matched, so default `mvn verify` ignores it
- **Only runs via dedicated Maven profile** (`-Pkarate`): activates a separate failsafe execution that includes `*KarateTest.java`

**Impact on existing workflows — zero interference:**

| Command / Script | Runs Karate? | Why |
|-----------------|-------------|-----|
| `mvn test` | No | surefire excludes `*KarateTest.java` |
| `mvn verify` | No | default failsafe includes only `*IT` + `*ComponentTest` |
| `scripts/verify-quick.sh` | No | runs `mvn test` |
| `scripts/final-check.sh` | No | runs `mvn verify` |
| `scripts/quality-check.sh` | No | runs `mvn verify` |
| `mvn verify -Pkarate` | **Caution** | Runs Karate but triggers full lifecycle — can kill a running app via devtools restart |
| `mvn failsafe:integration-test failsafe:verify -Pkarate` | **Yes (preferred)** | Direct goal invocation — no lifecycle, safe with running app |
| `scripts/karate-test.sh` | **Yes** | Convenience wrapper using direct goal invocation |

**Recommended profile:**

```xml
<profile>
    <id>karate</id>
    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-failsafe-plugin</artifactId>
                <configuration>
                    <includes>
                        <include>**/*KarateTest.java</include>
                    </includes>
                </configuration>
                <executions>
                    <execution>
                        <id>karate-tests</id>
                        <goals>
                            <goal>integration-test</goal>
                            <goal>verify</goal>
                        </goals>
                    </execution>
                </executions>
            </plugin>
        </plugins>
    </build>
</profile>
```

**Convenience script** (`scripts/karate-test.sh`):

```bash
#!/usr/bin/env bash
# karate-test.sh — Run Karate API tests against a running application.
# Usage: scripts/karate-test.sh [BASE_URL]
# Default BASE_URL: http://localhost:8080
# Requires: running flow-orchestrator application with SPRING_PROFILES_ACTIVE=local
set -euo pipefail
cd "$(dirname "$0")/../flow-orchestrator"
mvn -B failsafe:integration-test failsafe:verify -Pkarate -Dkarate.env="${KARATE_ENV:-local}" "$@"
```

**Important implementation note:** The script uses direct goal invocation (`failsafe:integration-test failsafe:verify`) instead of `mvn verify -Pkarate`. This is critical because `mvn verify` triggers the full lifecycle (compile, test, verify), which writes to `target/classes` and causes Spring Boot devtools to restart and kill the running application mid-test. Direct goal invocation skips lifecycle phases and runs only the Karate failsafe execution against already-compiled classes.

Run with: `scripts/karate-test.sh` (same pattern as `scripts/smoke-test.sh`)

### karate-config.js design

```javascript
function fn() {
  var env = karate.env || 'local';
  var config = {
    baseUrl: 'http://localhost:8080'
  };
  if (env === 'ci') {
    config.baseUrl = karate.properties['baseUrl'] || 'http://localhost:8080';
  }
  return config;
}
```

- No secrets in config file — PAT is injected via `application-local.yml` on the running app side, not on the test side
- Karate tests only need the base URL since authentication is handled by the app's GitLab client

### Test scenarios to implement

#### Feature 1: `issues-search.feature`

| Scenario | Method | Assertions |
|----------|--------|------------|
| Default search (no body) | `POST /api/issues/search` | status 200, `items` is array, `count` is number, `page` is number |
| Filtered search (state=opened) | `POST /api/issues/search` | status 200, items match filter criteria |
| Search with pagination | `POST /api/issues/search` | status 200, page matches request |
| `issueId` field present in results | `POST /api/issues/search` | each item contains `issueId` as number |

#### Feature 2: `issues-create.feature`

| Scenario | Method | Assertions |
|----------|--------|------------|
| Create with full payload | `POST /api/issues` | status 201, response has `id`, `issueId`, `title`, `state`, `labels` |
| Create with title only | `POST /api/issues` | status 201, `labels` is empty array |
| Create validation: blank title | `POST /api/issues` | status 400, `code` == `VALIDATION_ERROR` |
| Create validation: missing body | `POST /api/issues` | status 400 |

#### Feature 3: `issues-delete.feature`

| Scenario | Method | Assertions |
|----------|--------|------------|
| Delete existing issue | `DELETE /api/issues/{id}` | status 204, empty body |
| Delete non-existent issue | `DELETE /api/issues/999999` | status 404 |
| Delete validation: id=0 | `DELETE /api/issues/0` | status 400 |
| Delete validation: negative id | `DELETE /api/issues/-1` | status 400 |

#### Feature 4: `issues-lifecycle.feature` (the key scenario)

```gherkin
Feature: Issue lifecycle — create, fetch, delete

  Scenario: Full issue lifecycle
    # Step 1: Create
    Given url baseUrl
    And path 'api/issues'
    And request { title: 'Karate lifecycle test', description: 'Auto-created', labels: ['karate', 'smoke'] }
    When method post
    Then status 201
    And match response.issueId == '#number'
    And def createdIssueId = response.issueId

    # Step 2: Fetch (verify it appears in search)
    Given url baseUrl
    And path 'api/issues/search'
    And request { filters: { labels: ['karate'] } }
    When method post
    Then status 200
    And match response.items[*].issueId contains createdIssueId

    # Step 3: Delete
    Given url baseUrl
    And path 'api/issues', createdIssueId
    When method delete
    Then status 204

    # Step 4: Verify delete (should 404)
    Given url baseUrl
    And path 'api/issues', createdIssueId
    When method delete
    Then status 404
```

This single scenario replaces the most complex part of `smoke-test.sh` (the `extract_issue_id` + chained-delete flow) with clean, readable steps.

### What happens to smoke-test.sh

- **Keep it** during the transition as a fallback
- Once Karate tests are verified as equivalent and green, **deprecate** `smoke-test.sh`
- Update [local-quality-rules.instructions.md](../../.github/instructions/local-quality-rules.instructions.md) to reference `mvn verify -Pkarate` as the smoke-test command
- Long-term: remove `smoke-test.sh` entirely

### What happens to issues.http

- **Keep it** — it serves a different purpose (manual developer exploration in IntelliJ)
- It is not a test artifact and has no maintenance burden

## 7. Final Verdict

**Karate is an excellent fit** for replacing `scripts/smoke-test.sh`. The alignment is strong:

- **Tech stack match**: Java 21 + Maven + JUnit 5 — zero friction
- **Test philosophy match**: Real HTTP calls against a running application with real GitLab — same model as current smoke tests but with proper assertions
- **Quality improvement**: Deep JSON validation replaces grep-based field checking; structured reports replace stdout text
- **Constitution compliance**: No secrets in test code; externalized config via `karate-config.js`; tests clearly separated from unit/integration tiers
- **Low risk**: Single test-scope dependency; isolated in its own source root and Maven profile; existing tests untouched
- **High value**: The create → fetch → delete lifecycle scenario alone justifies the adoption — it replaces 30 lines of fragile bash with 20 lines of readable Gherkin

**Recommended action**: Proceed with implementation in the order defined in Section 6. Start with Slice 1 (framework setup), validate the build, then implement test features incrementally.
