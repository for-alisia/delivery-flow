## 1. Current Solution Analysis

### Architecture Overview

The project follows a layered architecture with four primary packages:

| Layer | Package | Purpose |
|---|---|---|
| **Orchestration** | `orchestration.issues` | Controllers, use-case services, ports, API models, mappers |
| **Domain** | `domain.issues` | Pure business models (`Issue`, `IssuePage`, `IssueQuery`, `IssueState`) |
| **Integration** | `integration.gitlab` | GitLab REST client adapter, response models, exception mapping |
| **Config** | `config` | `@ConfigurationProperties` records, validated at startup |
| **Common** | `common.errors`, `common.web` | Cross-cutting error types and global exception handler |

### Data Flow

```
HTTP POST /api/issues
  → IssuesController
    → GetIssuesService
      → IssuesRequestMapper.toIssueQuery() (API → domain model)
      → validatePerPage()
      → GetIssuesPort.getIssues() (port interface)
        → GitLabIssuesAdapter (implementation)
          → RestClient GET /projects/{path}/issues
          → GitLabIssueMapper (GitLab response → domain Issue)
    → IssuesResponseMapper.toResponse() (domain → API response)
  → IssuesResponse (returned to client)
```

### Technology Stack

- Java 21, Spring Boot 3.3.5, Spring Cloud 2023.0.3
- RestClient for outbound HTTP (despite Feign dependency being present)
- Lombok for boilerplate reduction (inconsistently applied)
- JaCoCo (85% minimum line coverage), Checkstyle, PMD, SpotBugs
- WireMock for component tests, MockRestServiceServer for adapter unit tests

### Test Structure

| Test Root | Naming | Count | Purpose |
|---|---|---|---|
| `src/test/java` | `*Test` | 7 | Unit tests with mocked collaborators |
| `src/test/integration/java` | `*IT` | 1 | `@WebMvcTest` slice tests |
| `src/test/component/java` | `*ComponentTest` | 1 | Full Spring Boot + WireMock |

## 2. Strong Sides

### 2.1 Clean Layered Architecture
The four-layer boundary (orchestration → domain ← integration, config) is properly enforced. Domain models are pure records with no framework annotations. Integration-layer response shapes (`GitLabIssueResponse`) never leak into orchestration or domain.

### 2.2 Ports and Adapters Pattern
`GetIssuesPort` is defined in orchestration and implemented by `GitLabIssuesAdapter` in integration. This makes the provider swappable and keeps orchestration code testable without real HTTP calls.

### 2.3 Immutable Data Models
All boundary models (`Issue`, `IssuePage`, `IssueQuery`, `IssuesRequest`, `IssuesResponse`, etc.) are Java records. Defensive copies are applied in compact constructors where lists are involved.

### 2.4 Configuration Validation
`GitLabProperties` and `IssuesApiProperties` use `@Validated` with constraint annotations. `IssuesApiProperties` includes a custom compact-constructor invariant check. The application fails fast on invalid config.

### 2.5 Centralized Exception Handling
`GlobalExceptionHandler` maps all exception types (`ValidationException`, `IntegrationException`, `MethodArgumentNotValidException`, `HttpMessageNotReadableException`, generic `Exception`) to sanitized `ErrorResponse` objects. No stack traces or internal details leak to clients.

### 2.6 Structured Error Taxonomy
`ErrorCode` enum provides semantic error categories. `IntegrationException` and `ValidationException` carry structured data rather than relying on message parsing.

### 2.7 Three-Tier Test Strategy
The project has a clear test separation: unit tests for logic, integration tests for HTTP contract, component tests for end-to-end flow with WireMock stubs. Support classes (`GitLabIssuesStubSupport`) keep component test setup clean.

### 2.8 Quality Toolchain
JaCoCo at 85%, Checkstyle, PMD, CPD, and SpotBugs are all integrated into the Maven verify lifecycle. Configuration files are repo-owned under `config/quality/`.

### 2.9 GitLab URL Handling
`GitLabProjectLocator` correctly parses project URLs, extracts API base URL, and handles namespaced project paths. The logic is tested for both valid and invalid inputs.

## 3. Gaps and Findings

### 3.1 — Unused Feign Dependency and Annotation

**Severity**: Medium  
**Files**: `pom.xml`, `FlowOrchestratorApplication.java`, `application.yml`

The project declares `spring-cloud-starter-openfeign` as a dependency, uses `@EnableFeignClients` on the application class, and configures `spring.cloud.openfeign.client.decode-slash` in YAML — yet no Feign client interface exists anywhere. The actual HTTP client is `RestClient`.

**Impact**: Unnecessary Spring Cloud BOM and transitive dependencies on the classpath. Confusing to new contributors who would expect to find Feign interfaces. Slower startup due to Feign auto-configuration scanning for non-existent interfaces.

**Recommendation**: Remove the Feign dependency, `@EnableFeignClients` annotation, and the `spring.cloud.openfeign` YAML config. See [Section 4](#4-restclient-vs-feign-client-analysis) for a detailed comparison.

---

### 3.2 — Insufficient Logging

**Severity**: Medium  
**Files**: `IssuesController.java`, `IssuesRequestMapper.java`, `IssuesResponseMapper.java`, `GitLabIssueMapper.java`, `GitLabProjectLocator.java`, `GitLabRestClientConfig.java`

Logging is present in `GetIssuesService`, `GitLabIssuesAdapter`, and `GlobalExceptionHandler`, but entirely absent from:

| Class | Missing Logging |
|---|---|
| `IssuesController` | No request-received or response-returned log. The entry point is silent. |
| `GitLabProjectLocator` | No startup log indicating the resolved API base URL and project path. Makes troubleshooting config issues harder. |
| `GitLabRestClientConfig` | No log confirming which base URL the RestClient was configured with. |
| `GitLabIssueMapper` | No debug-level log for mapping outcomes; tolerable but useful during integration debugging. |

Per the constitution (Principle 9): "Orchestration layer logs should record meaningful orchestration decisions and outcomes." The controller and service together represent the orchestration entry, but the controller produces zero log output.

**Recommendation**:
- Add `@Slf4j` to `IssuesController` and log at INFO level when a request is received and the response is returned (with count, page — no payload).
- Add startup INFO log in `GitLabProjectLocator` showing the resolved API base URL (no token).
- Add DEBUG-level logging in `GitLabIssueMapper` for mapping diagnostics when integration debugging is needed.

### 3.3 — Inconsistent Lombok Usage for Constructor Injection

**Severity**: Low–Medium  
**Files**: `IssuesRequestMapper.java`, `GitLabProjectLocator.java`

The code-guidance states: "`@RequiredArgsConstructor` is preferred for constructor injection when Lombok is used."

| Class | Pattern Used |
|---|---|
| `GitLabIssuesAdapter` | `@RequiredArgsConstructor` ✓ |
| `GetIssuesService` | `@RequiredArgsConstructor` ✓ |
| `IssuesController` | `@RequiredArgsConstructor` ✓ |
| `GlobalExceptionHandler` | `@Slf4j` ✓ (no deps to inject) |
| **`IssuesRequestMapper`** | **Manual constructor** ✗ |
| **`GitLabProjectLocator`** | **Manual constructor** ✗ |

`IssuesRequestMapper` defines a manual constructor for a single `final` field — a textbook `@RequiredArgsConstructor` case. `GitLabProjectLocator` performs logic in its constructor (URL resolution), which justifies a manual constructor — but it should be documented why Lombok is not used.

**Recommendation**:
- `IssuesRequestMapper`: Replace manual constructor with `@RequiredArgsConstructor`.
- `GitLabProjectLocator`: Keep the manual constructor (constructor logic), but add a brief comment explaining why `@RequiredArgsConstructor` is not used.
- Add `@Slf4j` to all classes that should log (see 3.2).

### 3.4 — `GetIssuesService` Takes API-Level Request Model

**Severity**: Medium  
**Files**: `GetIssuesService.java`

The service method signature is `getIssues(IssuesRequest request)` where `IssuesRequest` is an API-layer model from the `api` sub-package. This means the service depends on the HTTP request shape. If a second entry point (CLI, MCP, event listener) wants to call the same service, it must construct an `IssuesRequest` — an API concern.

Per the constitution (Principle 8): "Orchestration request models must never be passed directly into a domain service."

The `IssuesRequestMapper` is called inside the service, but the mapper returns a domain `IssueQuery`. The mapper should be the controller's responsibility, and the service should accept `IssueQuery` directly.

**Recommendation**: See [Section 5](#5-orchestration-package-structure-analysis) for the restructured flow.

### 3.5 — Redundant Defensive Copies in Record Accessors

**Severity**: Low  
**Files**: `GitLabIssueResponse.java`, `IssueFiltersRequest.java`

Both records create defensive copies in the compact constructor (e.g., `Collections.unmodifiableList(new ArrayList<>(labels))`) AND override the accessor methods to create a second copy:

```java
// Compact constructor already made it unmodifiable
labels = labels == null ? List.of() : Collections.unmodifiableList(new ArrayList<>(labels));

// Accessor creates ANOTHER copy
@Override
public List<String> labels() {
    return Collections.unmodifiableList(new ArrayList<>(labels));
}
```

The accessor override defeats the purpose of making the copy in the constructor. After the compact constructor runs, the field is already an unmodifiable list. Wrapping it again in `new ArrayList<>()` on every access is a waste.

**Recommendation**: Remove the overridden accessor methods. The compact constructor already guarantees immutability. If the concern is that someone could cast the inner list, use `List.copyOf()` in the constructor (which returns an unmodifiable list that cannot be unwrapped), and remove the accessor overrides entirely.

**Severity**: Low

**Severity**: Low

**Recommendation**: Remove the request-detail log from the adapter. The service owns the orchestration-level logging. The adapter should log only what is specific to the integration call (URL, response status, timing).

## 4. RestClient vs Feign Client Analysis

### The Question

The project declares `spring-cloud-starter-openfeign` as a dependency and `@EnableFeignClients` on the application class, but the actual HTTP client implementation uses Spring's `RestClient`. Should the project switch to Feign, or remove Feign and commit to RestClient?

### Recommendation: Remove Feign and keep RestClient

| Criterion | RestClient | Feign |
|---|---|---|
| **Spring alignment** | Part of Spring Framework 6.1+ (core). No additional dependency. | Requires Spring Cloud BOM and `spring-cloud-starter-openfeign`. |
| **Dependency footprint** | Zero additional dependencies beyond `spring-boot-starter-web`. | Adds Spring Cloud BOM (~20+ transitive artifacts), Feign core, Feign SLF4J, Feign OkHttp/Apache HC, Feign Form, etc. |
| **URI construction** | Full programmatic control via `UriBuilder`. Project-path encoding (`group%2Fproject`) and conditional query parameters are expressed naturally. | Declarative `@RequestLine` or `@GetMapping` with `@RequestParam`. Conditional query parameters require a `@QueryMap` map or multiple method overloads. Encoded path segments need `@PathVariable(encoded = true)` workarounds. |
| **Error handling** | `RestClientResponseException` hierarchy gives direct access to status code. Custom exception mapping (`GitLabExceptionMapper`) is straightforward. | Requires a custom `ErrorDecoder` bean. Error decoder receives a Feign `Response` object, not a Spring exception. Requires extra mapping to align with the project's `IntegrationException` model. |
| **Testability** | `MockRestServiceServer` (in-process, no port needed) works out of the box. Already used in `GitLabIssuesAdapterTest`. | Requires WireMock or a similar stub server for unit-level adapter tests. `MockRestServiceServer` does not work with Feign. |
| **Pagination / response headers** | Full access to `ResponseEntity<T>` including headers (e.g., GitLab's `X-Total`, `X-Next-Page` headers for future pagination). | Possible via custom `Decoder` or `ResponseInterceptor`, but more boilerplate than RestClient's built-in `toEntity()`. |
| **Architecture fit** | Sits inside the adapter (`GitLabIssuesAdapter`) as an injected collaborator. The adapter owns all URL construction and mapping. Clean separation. | Declarative interface becomes the adapter itself. Works well for simple CRUD but resists the complex conditional URI building this project needs. |
| **Future direction** | `RestClient` is Spring's strategic replacement for `RestTemplate`. Active development and long-term support. | Spring Cloud OpenFeign has been in maintenance mode. Spring team recommends `RestClient` or `WebClient` for new projects. |

### Why Feign is not a good fit here

1. **Complex URI construction**: GitLab API requires URL-encoded project paths in the URI (`/projects/group%2Fsubgroup%2Fproject/issues`), conditional query parameters (state, labels, assignee, milestone), and pagination params. RestClient's `UriBuilder` handles this naturally. Feign's declarative approach would require `@QueryMap` or multiple method signatures.

2. **Pagination headers**: GitLab returns pagination metadata (`X-Total`, `X-Next-Page`, etc.) in response headers. RestClient gives direct access via `toEntity()`. Feign requires a custom `Decoder` to access response headers.

3. **Already working and tested**: The RestClient implementation is clean, tested, and production-ready. Switching to Feign would be an unnecessary rewrite.

4. **Dependency bloat**: The Spring Cloud BOM adds significant transitive dependencies for a feature that is not used and would not improve the codebase.

### Action: Remove all Feign artifacts
- Remove `spring-cloud-starter-openfeign` from `pom.xml`
- Remove `spring-cloud-dependencies` BOM from `dependencyManagement`
- Remove `spring-cloud.version` property from `pom.xml`
- Remove `@EnableFeignClients` from `FlowOrchestratorApplication.java`
- Remove `spring.cloud.openfeign` section from `application.yml`

## 5. Orchestration Package Structure Analysis

### Current Structure

```
orchestration/
  issues/
    GetIssuesPort.java            ← Port interface (defines integration contract)
    GetIssuesService.java         ← Use-case service (takes IssuesRequest, delegates mapping + validation)
    api/
      IssuesController.java       ← REST controller (thin, delegates to service)
      IssuesRequest.java          ← API request record
      IssuesResponse.java         ← API response record
      IssueFiltersRequest.java    ← Filter sub-model
      IssueResponseItem.java      ← Response item sub-model
      PaginationRequest.java      ← Pagination sub-model
      IssuesRequestMapper.java    ← Maps API request → domain IssueQuery
      IssuesResponseMapper.java   ← Maps domain IssuePage → API response
```

### Current Flow Logic

1. `IssuesController.getIssues(IssuesRequest)` receives the HTTP POST body (nullable).
2. Controller delegates directly to `GetIssuesService.getIssues(IssuesRequest)`.
3. **Inside the service**: `IssuesRequestMapper.toIssueQuery(request)` converts API request to domain `IssueQuery`.
4. **Inside the service**: `validatePerPage(query.perPage())` checks business constraint.
5. **Inside the service**: `getIssuesPort.getIssues(query)` calls the integration adapter.
6. Controller calls `IssuesResponseMapper.toResponse(issuePage)` to map the result.

### Problems with Current Approach

**Problem 1: Service depends on API model**

`GetIssuesService` accepts `IssuesRequest` (an HTTP API concern). This means:
- The service cannot be called from a non-HTTP entry point without constructing an HTTP request model.
- The service has a dependency on the `api` sub-package — an inward dependency violation.
- The `IssuesRequestMapper` (API → domain mapping) is called inside the service, but per Principle 8 of the constitution, "mapping responsibility belongs to the layer that owns the transformation." The API-to-domain mapping belongs at the API layer (controller), not inside the service.

**Problem 2: Asymmetric mapping responsibility**

Request mapping happens inside the service, but response mapping happens in the controller. This is inconsistent. Both mappings are API-boundary concerns and should happen at the same location.

**Problem 3: Service combines mapping delegation with validation**

The service method does three things: map, validate, call port. The mapping should be done before the service is called. The validation is a legitimate service concern.

### Recommended Structure

```
orchestration/
  issues/
    GetIssuesPort.java            ← Port interface (unchanged)
    GetIssuesService.java         ← Service accepts IssueQuery (domain model), validates, delegates
    api/
      IssuesController.java       ← Maps request → IssueQuery, calls service, maps response
      IssuesRequest.java          ← API request record (unchanged)
      IssuesResponse.java         ← API response record (unchanged)
      IssueFiltersRequest.java    ← Filter sub-model (unchanged)
      IssueResponseItem.java      ← Response item sub-model (unchanged)
      PaginationRequest.java      ← Pagination sub-model (unchanged)
      IssuesRequestMapper.java    ← Maps API request → domain IssueQuery (unchanged)
      IssuesResponseMapper.java   ← Maps domain IssuePage → API response (unchanged)
```

### Recommended Flow

```
IssuesController:
  1. IssueQuery query = issuesRequestMapper.toIssueQuery(request)    ← mapping at API boundary
  2. IssuePage page = getIssuesService.getIssues(query)              ← service accepts domain model
  3. return issuesResponseMapper.toResponse(page)                     ← mapping at API boundary

GetIssuesService:
  1. validatePerPage(query.perPage())                                 ← business validation
  2. return getIssuesPort.getIssues(query)                            ← port call
```

This change:
- Makes both mappers the controller's responsibility (symmetric).
- Makes the service reusable from any entry point that can construct an `IssueQuery`.
- Removes the service dependency on the `api` package.
- Follows the constitution's mapping rules (Principle 8).

## Revision 2 — Domain Architecture Flaw: GitLab-Specific Models in Domain Layer

**Date**: 2026-04-01  
**Severity**: **CRITICAL — Architectural Violation**  
**Scope**: `domain.issues` package, `GetIssuesPort` contract, `GitLabIssuesAdapter` dependency direction

This revision supersedes the following earlier assessments that incorrectly praised the domain design:
- **Section 2.1** claimed "Domain models are pure records with no framework annotations" — technically true, but the models themselves should not exist in domain at all for this feature.
- **Section 2.1** claimed "Integration-layer response shapes never leak into orchestration or domain" — the leak goes the other direction: domain shapes were defined to match GitLab's API, then imported by integration.

### The Problem

The `domain.issues` package contains four types that are GitLab-specific pass-through shapes masquerading as domain concepts:

| Type | What It Actually Is | Why It Doesn't Belong in Domain |
|---|---|---|
| `Issue` | Mirror of GitLab's issue response shape (id, title, description, state, labels, assignee, milestone, parent) | "Issue" is GitLab vocabulary. The domain has no business logic that operates on this shape. |
| `IssuePage` | Pagination wrapper (items, count, page) | Pagination is a transport/API mechanic — explicitly forbidden in domain by Principle 3. |
| `IssueQuery` | Pagination + GitLab-specific filter parameters (page, perPage, state, label, assignee, milestone) | Pagination metadata and provider-specific filter names are transport concerns. |
| `IssueState` | Enum with GitLab-specific string values (`opened`, `closed`, `all`) | These are GitLab API parameter values, not domain-meaningful business states. |

The current "get issues" feature is a **pure pass-through**: it takes issues from the GitLab API and returns them to the client. There is no business logic involved — no transformation, no calculation, no decision-making. A pass-through feature should not touch the domain layer at all.

### Constitution Violations

**Principle 1 — Integration must not depend on domain**:
- `GitLabIssuesAdapter` imports `domain.issues.Issue`, `domain.issues.IssuePage`, `domain.issues.IssueQuery`
- `GitLabIssueMapper` imports `domain.issues.Issue`
- This is a hard `[MUST]` violation of the dependency boundary

**Principle 2 — Integration adapters must map to orchestration contracts**:
- The constitution states: "Integration adapters must map provider-specific responses to **orchestration contracts** before returning data."
- Currently, the adapter maps GitLab responses to **domain types**, not orchestration contracts.

**Principle 3 — Domain models must not carry transport concerns**:
- `IssuePage` carries `count` and `page` — pagination metadata
- `IssueQuery` carries `page` and `perPage` — pagination parameters
- `IssueState.value()` returns GitLab API string values — provider-specific field encoding
- The constitution states: "no provider-specific field names, no pagination metadata"

**Principle 3 — If a concept has business meaning, it belongs in domain. If it represents transport, presentation, or external API mechanics, it does not**:
- The "get issues" feature is transport/API mechanics: relay data from one API to another
- No business meaning is derived, transformed, or enforced by the domain

### Correct Architecture

#### For pass-through features (current "get issues"):

```
HTTP POST /api/issues
  → IssuesController (orchestration.issues.api)
    → IssuesRequestMapper.toIssueQuery()     ← maps API request → orchestration-level query
    → GetIssuesService (orchestration.issues)
      → GetIssuesPort.getIssues()            ← port defined in orchestration with orchestration types
        → GitLabIssuesAdapter (integration)
          → RestClient GET → GitLabIssueResponse     ← provider-specific
          → maps to orchestration-level types          ← adapter maps to orchestration contract
    → IssuesResponseMapper.toResponse()      ← maps orchestration types → API response
```

Domain layer: **not involved**. The port contract types (`IssueQuery`, `IssuePage`, `Issue`) move from `domain.issues` to `orchestration.issues` — they are orchestration-level contracts, not domain models.

#### For future business features (aging WIP calculation):

```
orchestration                          domain                    integration
    │                                     │                          │
    │  WorkItemAgeService                 │  WorkItem (agnostic)     │
    │    → fetch from port ───────────────┼──────────────────────────→ GitLabAdapter
    │    → map to domain WorkItem ────────→                          │ maps GitLab → orch contract
    │    → call domain age calculator ────→  AgingCalculator         │
    │    ← domain result ←───────────────←  AgeReport               │
    │    → map to API response            │                          │
```

Domain models like `WorkItem` would be **provider-agnostic** — no "issue", no "merge request", no GitLab vocabulary. The domain would operate on business-meaningful concepts (work items, aging, WIP limits) that apply regardless of whether the backing provider is GitLab, GitHub, Jira, or Azure DevOps.

### Dependency Direction Fix

**Current (broken)**:
```
orchestration.issues ──→ domain.issues ←── integration.gitlab.issues
                              ↑
                     [VIOLATION: integration depends on domain]
```

**Correct for pass-through**:
```
orchestration.issues.api ──→ orchestration.issues ←── integration.gitlab.issues
                                     ↑
                          [port + contract types live here]
                          [domain is not involved]
```

### Recommended Actions

#### Immediate — Relocate pass-through types to orchestration

1. Move `Issue`, `IssuePage`, `IssueQuery`, `IssueState` from `domain.issues` to `orchestration.issues` (these are orchestration-level contracts for the pass-through feature).
2. Delete the empty `domain.issues` package.
3. Update `GitLabIssuesAdapter` and `GitLabIssueMapper` imports to reference `orchestration.issues` types instead of `domain.issues`.
4. Update all tests accordingly.
5. Run `mvn clean verify` — all quality gates must pass.

#### Future — When business logic is added

6. Create true domain models (`WorkItem`, etc.) in `domain` that are provider-agnostic — no GitLab vocabulary, no pagination, no provider-specific state values.
7. Orchestration maps between integration contracts and domain models when business logic is needed.
8. Pass-through features continue to bypass domain entirely.

### Impact Assessment

This is not a code defect — the application functions correctly. It is a **structural violation** of three constitution principles that will compound as more features are added. If left unaddressed:
- Every new pass-through feature will add more provider-specific shapes to domain
- Domain will become a mirror of GitLab's API instead of a business logic layer
- When a second provider (GitHub, Jira) is added, the domain will be locked to GitLab's vocabulary
- True business logic will have nowhere clean to live
