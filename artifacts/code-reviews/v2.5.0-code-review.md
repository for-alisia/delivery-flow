### Request Flow for `PATCH /api/issues/{issueId}`

```
IssuesController.updateIssue()
  ├── @Positive validates issueId path variable
  ├── IssuesRequestValidator.validateUpdateRequest()
  │     ├── hasEffectiveChange() — at-least-one-field guard
  │     ├── validateOptionalTitle() — blank + length via IssuesApiValidationProperties
  │     ├── validateDescription() — max length
  │     ├── validateLabels() — count + blank/null per element (addLabels + removeLabels)
  │     └── validateOverlappingLabels() — set intersection rejection
  ├── IssuesRequestMapper.toUpdateIssueInput() — DTO → orchestration model
  ├── IssuesService.updateIssue()
  │     └── IssuesPort.updateIssue()
  │           └── GitLabIssuesAdapter.requestIssueUpdate()
  │                 ├── GitLabIssuesMapper.toUpdateRequest() — model → GitLab wire DTO
  │                 │     title, description, addLabels→comma-string, removeLabels→comma-string
  │                 ├── RestClient.put() to /projects/{id}/issues/{iid}
  │                 └── GitLabIssuesMapper.toIssue() — GitLab response → IssueSummary
  └── IssuesResponseMapper.toIssueSummaryDto() — model → response DTO
```

### Observations on Flow Structure

The update flow follows the same well-established pattern as create: thin controller → validator → mapper → service → port → adapter → mapper roundtrip. The validator is properly shared between create and update, with the key difference being `validateOptionalTitle()` vs `validateRequiredTitle()`. The configurable limits (`IssuesApiValidationProperties`) are correctly reused across both endpoints, satisfying the locked constraint about shared limits.

## 3. Strong Sides

### 3.1 Clean Layer Separation

The update feature strictly follows the ports-and-adapters pattern. `UpdateIssueInput` is the orchestration model, `UpdateIssueRequest` is the REST DTO, and `GitLabUpdateIssueRequest` is the integration wire DTO. Each has its own mapper. No layer leakage was observed. Constitution Principles 1, 2, and 8 are satisfied.

### 3.2 Shared Validation Configuration

`IssuesApiValidationProperties` centralizes title length, description length, and label count limits. Both `validateCreateRequest()` and `validateUpdateRequest()` draw from the same configuration. This was a locked requirement and is correctly implemented. The configuration is validated at startup with clear error messages (`IssuesApiProperties` compact constructor, `IssuesApiValidationProperties.requirePositive()`).

### 3.3 Consistent Immutability

All new records use defensive copies:
- `UpdateIssueRequest` — copies and wraps `addLabels`/`removeLabels` in `Collections.unmodifiableList`
- `UpdateIssueInput` — normalizes null to `List.of()`, copies via `List.copyOf()`
- `GitLabUpdateIssueRequest` — plain record with `@JsonInclude(NON_NULL)` for clean wire format

This satisfies Constitution Principle 6 and the Defensive Copy Gate.

### 3.4 Comprehensive Validation

The `IssuesRequestValidator.validateUpdateRequest()` method covers all locked constraints:
- At-least-one-field guard
- Optional title: blank, min/max length
- Description: max length, empty string allowed
- Label counts per field
- Null/blank per-element label validation with indexed error messages
- Overlapping add/remove label rejection

Validation errors are aggregated — multiple violations in one request produce a complete error response rather than failing on the first one.

### 3.5 Integration Wire Format

`GitLabUpdateIssueRequest` correctly uses `@JsonProperty("add_labels")` and `@JsonProperty("remove_labels")` for the snake_case GitLab API contract. Labels are sent as comma-separated strings via `toCommaSeparatedLabels()`. The `@JsonInclude(NON_NULL)` annotation ensures only changed fields are sent to GitLab, which is essential for a PATCH-like semantic where absent means "no change."

### 3.6 Test Coverage Quality

The update feature has strong multi-layer test coverage:
- **Unit**: `IssuesServiceTest` — delegation + error propagation (2 tests)
- **Unit**: `IssuesRequestValidatorTest` — 6 update-specific tests covering edge cases
- **Unit**: `GitLabIssuesMapperTest` — 2 update mapping tests (label concat, null handling)
- **Unit**: `GitLabIssuesAdapterTest` — 3 update tests (happy path, error mapping, transport failure)
- **Unit**: `IssuesRequestMapperTest` — 2 update mapping tests
- **Integration**: `IssuesControllerIT` — 3 update IT tests (happy path, empty payload rejection, overlapping labels)
- **Component**: `IssuesApiComponentTest` — 5 update component tests (full flow, clear description, 404, 500, overlap rejection)
- **Karate**: `issues-update.feature` — 2 smoke scenarios (full update + validation error)

### 3.7 Documentation Updates

`documentation/capabilities/issues.md` is properly updated with `UpdateIssueInput`, `UpdateIssueRequest`, `GitLabUpdateIssueRequest`, and the adapter's update role. The `.http` file includes PATCH examples covering the main use cases.

### 3.8 Logging

The update flow has consistent structured logging:
- Controller: entry with field presence summary, exit with issueId
- Service: entry with effective field count and label counts, exit with issueId
- Adapter: before and after GitLab call with issueId
- Mapper: debug-level field mapping summary

This satisfies the Logging Gate requirements.

## 4. Findings and Gaps

### 4.1 MEDIUM — `enrichIssue()` manual builder-copy anti-pattern persists

**What is happening:** `IssuesService.enrichIssue()` (line ~140) manually copies all 9 fields of `IssueSummary` into a new builder just to replace `changeSets`. This was identified in v2.4.0 analysis as a fragility risk.

**File:** [flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java#L137-L149)

**Why it is a problem:** When `IssueSummary` gains new fields (which it will — e.g., `dueDate`, `weight`, `webUrl`), `enrichIssue()` will silently drop them. The method has no compile-time safety net. This is exactly the "breaks silently when fields are added" pattern flagged in `code-guidance.md` → Model Structure Gate.

**What to prefer:** Add a `toBuilder()` method to `IssueSummary` (Lombok `@Builder(toBuilder = true)`) and replace the method with:
```java
private IssueSummary enrichIssue(IssueSummary issue, List<ChangeSet<?>> changeSets) {
    return issue.toBuilder().changeSets(changeSets).build();
}
```

**Expected benefit:** Field additions to `IssueSummary` are automatically propagated. The enrichment method becomes a one-liner that cannot drift.

### 4.2 LOW — `IntStream.range` pattern in `enrichWithLabelEvents`

**What is happening:** `IssuesService.enrichWithLabelEvents()` uses `IntStream.range(0, issuePage.items().size()).mapToObj(index -> enrichIssue(...))` to zip issues with their label events by position.

**File:** [flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java#L128-L136)

**Why it is a problem:** The `IntStream.range` pattern is unidiomatic for paired-element iteration. It obscures intent and is fragile if either list's ordering assumption changes. The code-guidance Collections And Streams Gate says "streams are not used when a loop is simpler and clearer."

**What to prefer:** A simple indexed for-loop with explicit size guard, or a utility zip if one is added later. This is low priority since correctness is not affected and the current size (3 lines) is manageable.

**Expected benefit:** Marginally clearer intent. Easier for future maintainers to reason about the index relationship.

### 4.3 MEDIUM — `context-map.md` Capability Index not updated for PATCH endpoint

**What is happening:** The Capability Index in [documentation/context-map.md](../../documentation/context-map.md#L33) lists the issues capability as:
> `issues — search, create, delete, get-single, label-event history`

with endpoints `POST /api/issues/search`, `POST /api/issues`, `DELETE /api/issues/{issueId}`, `GET /api/issues/{issueId}`. The PATCH endpoint and "update" capability are missing.

**Why it is a problem:** Constitution Principle 11 requires documentation to stay in sync with implementation. The context-map is the primary capability index read by all agents. Missing the update endpoint means future planning may not account for it or may re-implement it.

**What to prefer:** Update the capability description to include "update" and add `PATCH /api/issues/{issueId}` to the endpoint list.

**Expected benefit:** Agents and contributors discover the update capability from the index without needing to read the full issues capability document.

### 4.4 LOW — Logging gap in `updateIssue` controller: validation failure path is silent

**What is happening:** In `IssuesController.updateIssue()`, when `issuesRequestValidator.validateUpdateRequest()` throws `ValidationException`, the controller's entry log fires but there is no specific log for the validation rejection at the controller level. The `GlobalExceptionHandler` logs a generic `warn` with detail count, but the `issueId` from the request is lost by the time it reaches the exception handler.

**Why it is a problem:** For update operations, the `issueId` is diagnostic context that should survive into the validation failure log. Without it, correlating a validation error to a specific update target requires tracing through other means.

**What to prefer:** This is mitigated by the fact that the controller's entry log already contains `issueId`. The two log lines together provide a correlation path. No action required unless a structured correlation ID is introduced.

**Expected benefit:** N/A — informational observation.

### 4.5 MEDIUM — Missing test for duplicate labels within the same field

**What is happening:** The user story notes (in run log) identify a bug: "It's possible to send duplicated labels inside addLabels or removeLabels, for example: `addLabels: ['label', 'label']`." The `IssuesRequestValidator` does not reject duplicates within `addLabels` or `removeLabels`. The `.http` example at line 133 even demonstrates this: `"removeLabels": ["bug", "bug"]`.

No test exists to document this behavior (whether it should pass or fail).

**Why it is a problem:** Sending `addLabels: ["bug", "bug"]` will produce the GitLab wire value `"add_labels": "bug,bug"`. GitLab handles this gracefully (it adds the label once), but it indicates either:
1. A client error that should be caught with a clear validation message, OR
2. An intentional "lenient" behavior that should be documented and tested

Currently it is neither validated nor tested — it falls into an undocumented gray zone.

**What to prefer:** Add a deduplication step in `IssuesRequestValidator.validateLabels()` or in `IssuesRequestMapper.toUpdateIssueInput()`. If choosing to be lenient, add a unit test documenting that duplicates are silently accepted and produce a single GitLab label operation. Either way, make the behavior explicit.

**Expected benefit:** Eliminates ambiguity. Prevents unnecessary GitLab API calls with redundant label operations.

### 4.6 LOW — Missing PATCH-specific `issueId` boundary ITs for 0/negative values

**What is happening:** The integration test `IssuesControllerIT` validates `@Positive` rejection for `issueId` on DELETE and GET endpoints but has no equivalent test for the PATCH endpoint with `issueId=0` or `issueId=-1`.

**File:** [flow-orchestrator/src/test/integration/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/IssuesControllerIT.java](../../flow-orchestrator/src/test/integration/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/IssuesControllerIT.java)

**Why it is a problem:** The `@Positive` annotation on the PATCH path variable is the same mechanism as DELETE/GET, and Spring's `ConstraintViolationException` handling is tested elsewhere. The risk is very low. However, test-instructions require "cover edge cases relevant to the change," and this is an edge case for the new PATCH endpoint.

**Expected benefit:** Explicit proof that the PATCH `@Positive` annotation works end-to-end. Prevents regression if someone accidentally removes the annotation.

### 4.7 LOW — `UpdateIssueRequest` does not sanitize null elements in label lists

**What is happening:** `CreateIssueRequest`'s compact constructor filters null elements via `.stream().filter(Objects::nonNull).toList()` before copying. `UpdateIssueRequest`'s compact constructor copies elements as-is via `new ArrayList<>(addLabels)`, preserving null elements.

**Files:**
- [CreateIssueRequest.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/dto/CreateIssueRequest.java#L14-L17)
- [UpdateIssueRequest.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/dto/UpdateIssueRequest.java#L14-L19)

**Why it is a problem:** The behavioral inconsistency is minor because the `IssuesRequestValidator.validateLabels()` already catches null elements at index level with a clear error message (`addLabels[1] must not be null`). However, the different defensive strategies between two sibling DTOs is a maintenance inconsistency. If a future consumer bypasses validation, `UpdateIssueRequest` preserves nulls while `CreateIssueRequest` silently strips them.

**What to prefer:** Align the compact constructors: either both filter nulls or neither does (relying on validation). The current approach of validator-catches-nulls is correct and tested, so the impact is cosmetic.

**Expected benefit:** Consistent defensive strategy across sibling DTOs.

### 4.8 LOW — `.http` example comment references uppercase `LABEL` for get-single

**What is happening:** The `.http` file comment at line 164 says: `change[field=LABEL (uppercase enum)]` for the GET single-issue response. But `ChangeField.LABEL` serializes as `"label"` (lowercase) via `@JsonValue`. The comment is misleading.

**File:** [flow-orchestrator/http/issues.http](../../flow-orchestrator/http/issues.http#L164)

**Why it is a problem:** Developers using the `.http` file as a reference may expect uppercase `LABEL` in the response. The wire format is `"label"` (lowercase), which is correct per `ChangeField.value()` and confirmed by tests.

**What to prefer:** Update the comment to reflect the actual wire format: `change[field="label" (lowercase wire value)]`.

## 6. Final Verdict

The `update-issue-api` feature is well-implemented. It follows the established architectural patterns, has comprehensive multi-layer test coverage, clean layer separation, and proper wire-format handling. The shared validation configuration between create and update endpoints is correctly implemented per the locked requirements.

**Strengths:**
- Clean ports-and-adapters adherence for the new PATCH flow
- Proper configurable validation shared between create and update
- Strong test coverage across 5 test layers (unit, mapper, adapter, integration, component, Karate)
- Correct `@JsonInclude(NON_NULL)` on the GitLab wire DTO for partial-update semantics
- Documentation updated in `capabilities/issues.md` and `.http` examples

**Remaining risks:**
- The `enrichIssue()` manual builder-copy (R-2) is the most significant maintenance risk. It will silently drop fields when `IssueSummary` grows. This was flagged in v2.4.0 and remains unaddressed.
- The duplicate-labels-in-same-field gap (R-3) is a minor functional ambiguity that should be resolved with an explicit decision.
- The `context-map.md` index gap (R-1) should be fixed immediately as it affects agent planning accuracy.

**Overall assessment:** The codebase is in good health. The v2.4.0 refactoring (shared models, DTO consolidation, type-safe change hierarchy) provides a solid foundation that the update feature builds on cleanly. The findings above are improvement opportunities, not blockers. No architectural violations or security concerns were observed.
