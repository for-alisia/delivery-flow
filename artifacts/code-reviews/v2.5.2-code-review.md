## 2. Current architecture and request flow

### Current structure

The `orchestration` package currently holds the right responsibilities for this feature.

- `orchestration/milestones/rest/MilestonesController.java` is the HTTP boundary.
- `orchestration/milestones/rest/MilestonesRequestValidator.java` owns boundary rules that are checked before orchestration logic runs.
- `orchestration/milestones/rest/mapper/MilestonesRequestMapper.java` and `MilestonesResponseMapper.java` isolate transport mapping.
- `orchestration/milestones/MilestonesService.java` coordinates the use case and stays provider-agnostic through `MilestonesPort.java`.
- `integration/gitlab/milestones/GitLabMilestonesAdapter.java` owns URI construction, outbound GitLab calls, and provider DTO mapping through `GitLabMilestonesMapper.java`.

This matches the constitution well: thin entry point, service-orchestrated use case, port/adaptor split, and sanitized exception translation at the shared web boundary.

### Request flow for `POST /api/milestones`

1. `MilestonesController.createMilestone()` accepts the REST request.
2. `MilestonesRequestValidator.validateCreateRequest()` enforces title rules, date parsing, and cross-field ordering.
3. `MilestonesRequestMapper.toCreateMilestoneInput()` maps the REST DTO to the orchestration input.
4. `MilestonesService.createMilestone()` logs the use-case start and delegates to `MilestonesPort.createMilestone()`.
5. `GitLabMilestonesAdapter.createMilestone()` posts to `/projects/{projectPath}/milestones` using the configured project from `GitLabProjectLocator`.
6. `GitLabOperationExecutor` and `GitLabExceptionMapper` sanitize provider failures into `IntegrationException` categories.
7. `GitLabMilestonesMapper.toMilestone()` maps GitLab `iid` to orchestration `milestoneId`.
8. `MilestonesResponseMapper.toCreateMilestoneResponse()` returns only `milestoneId` and `title` with HTTP `201`.

### What is working well

- The create and search flows reuse the same milestone capability package without leaking GitLab DTOs into orchestration.
- The configured-project rule from the story is respected: no project identifier is accepted from the client request.
- Error handling follows the existing issues pattern and stays centralized in `GlobalExceptionHandler`.

### What would be cleaner if refactored

The overall milestone package should not be reorganized broadly. The next cleaner step is narrower:

- Split `MilestonesRequestValidator` into create-specific and search-specific validators if more milestone operations are added.
- Promote create dates from raw strings to typed `LocalDate` values in the orchestration model when the capability grows, so parsing happens once at the boundary and provider formatting stays in the adapter mapper.

## 4. Findings and gaps

### 4.1 HIGH — The milestone-create smoke verification is not repeatable in the current environment

Status: FAIL

Observed facts:

- `scripts/karate-test.sh` exited with code `1`.
- `flow-orchestrator/src/test/karate/resources/milestones/milestones-create.feature` uses fixed titles: `Release v1.0` and `Q2 2026 Delivery`.
- During the same smoke run, the search smoke response already contained milestones with titles `Release v1.0` and `Q2 2026 Delivery`.
- The two successful-create scenarios in `milestones-create.feature` expected `201` but received `502`.

Why this is a problem:

- The required API smoke gate is currently red, so this version cannot be considered review-pass.
- The running-app verification for the write path is stateful and non-repeatable. It is currently impossible to tell from the smoke suite alone whether the problem is implementation logic or persistent-project data collision.

What to prefer instead:

- Generate unique milestone titles per smoke-test run.
- Run the create smoke against a disposable test project, or add cleanup that removes created milestones if the environment allows it.
- Keep the invalid-create scenario static; only the successful create scenarios need unique test data.

Expected benefit:

- Stable end-to-end verification of the real write path.
- Clearer separation between code defects and environment/data collisions.

Reasoned conclusion:

The most plausible cause of the current smoke failure is persistent-state collision in the configured GitLab project rather than a missing controller/service/adapter path, because the same run shows those exact titles already exist and all isolated tests for the create path are green.

### 4.2 MEDIUM — Top-level capability documentation is behind the implementation

Status: FAIL

Observed facts:

- `documentation/context-map.md` still describes milestones as `milestone search and shared milestone contracts` and lists only `POST /api/milestones/search`.
- `documentation/project-overview.md` still says milestone search is the implemented milestone capability.
- `documentation/capabilities/milestones.md` and `flow-orchestrator/http/milestones.http` already describe `POST /api/milestones` correctly.

Why this is a problem:

- The top-level discovery docs are the first routing surface for future contributors and agents.
- The current drift violates the project rule that integration documentation and capability maps stay synchronized with shipped behavior.

What to prefer instead:

- Update `documentation/context-map.md` and `documentation/project-overview.md` in the same change set that ships the feature.

Expected benefit:

- Accurate capability discovery.
- Less planning and review friction for later milestone work.

### 4.3 MEDIUM — Create request validation follows the existing issues pattern, but not the repository’s own validation guidance

Status: FAIL

Observed facts:

- `CreateMilestoneRequest.java` has no bean-validation annotations.
- `MilestonesController.java` still uses `@Valid` on the request boundary.
- `MilestonesRequestValidator.java` manually enforces blank-title and title-length rules that are declarative and could live on the DTO.
- `IssuesRequestValidator.java` follows the same manual style, so the code is consistent with the current house pattern but not with `documentation/code-guidance.md`.

Why this is a problem or risk:

- The create boundary contract is less self-describing than it should be.
- `@Valid` contributes almost nothing for the create title today, which makes the controller signature look stricter than the DTO actually is.
- The project guidance explicitly says declarative rules should use annotations when they can be expressed cleanly.

What to prefer instead:

- Put `@NotBlank` and `@Size(min = 4, max = 499)` on `CreateMilestoneRequest.title`.
- Keep `MilestonesRequestValidator` only for ISO date parsing and the cross-field `dueDate after startDate` rule.

Expected benefit:

- A clearer REST contract.
- Less manual validation code.
- Better alignment with the repository’s stated boundary-validation rules.

### 4.4 LOW — Logging is mostly present, but the create path omits useful non-sensitive request context

Status: FAIL

Observed facts:

- `MilestonesService.java` logs create start and completion.
- `GitLabMilestonesAdapter.java` logs outbound create intent and completion.
- `GitLabOperationExecutor.java` logs sanitized status/category for provider failures.
- `MilestonesController.java`, `MilestonesService.java`, and `GitLabMilestonesAdapter.java` only log `titleLength`, `descriptionPresent`, `startDatePresent`, and `dueDatePresent` for create requests.

Why this is a problem or risk:

- The current logs do not expose enough safe business context to diagnose why one create request failed while another succeeded.
- In the current smoke failure, the create-side logs would not show the non-sensitive date values being sent.

What to prefer instead:

- Log `startDate` and `dueDate` directly.
- Keep description redacted.
- If the team does not want full titles in logs, log a safe title surrogate such as a deterministic hash or other non-sensitive identifier.

Expected benefit:

- Faster diagnosis of live provider rejections without exposing secrets or full payloads.

## 5. Prioritized refactoring plan

1. Fix the milestone-create smoke test first. Make the happy-path Karate create titles unique per run, or run them against a disposable GitLab project. Re-run `scripts/karate-test.sh` until milestone create passes green.
2. Synchronize the top-level capability docs. Update `documentation/context-map.md` and `documentation/project-overview.md` so milestone creation is discoverable from the same places that already route users to milestone search.
3. Align create-title validation with `code-guidance.md`. Move simple title rules onto `CreateMilestoneRequest.java` and leave only cross-field/date parsing in `MilestonesRequestValidator.java`.
4. Strengthen create-flow logging. Keep the current sanitized error mapping, but log safe request context such as `startDate` and `dueDate` so live failures are diagnosable.
5. Optional next cleanup when the capability expands: type create dates as `LocalDate` in orchestration models and split combined create/search validator responsibilities.

## 6. Final verdict

The user story is largely implemented in code. The endpoint exists, the configured-project routing rule is respected, title and date validation rules are present, the response returns `milestoneId` from GitLab `iid`, and provider failures are sanitized through the established error pattern. The orchestration structure is currently clean: controller at the boundary, mapper and validator at the REST edge, service for use-case coordination, and GitLab adapter isolated behind a port.

The solution is not review-pass in its current state because the required running-app smoke verification is failing. `scripts/karate-test.sh` is red on the two successful milestone-create scenarios. The strongest evidence points to a non-repeatable smoke design that uses fixed titles against a persistent GitLab-backed project, not to a missing implementation path, but the result is the same for review: the feature cannot be signed off while the create smoke gate is failing.

Explicit answers to the required review questions:

- Is logging sufficient? No. Logging is structurally present and mostly correct, but the create flow lacks enough safe request context. `startDate` and `dueDate` should be logged directly; description should stay redacted.
- Is the code following consistent patterns and instructions? Mostly yes for layering and ports/adapters, partially no for repository guidance. The implementation follows the existing issues-capability pattern, but the create boundary does not follow `code-guidance.md`'s preference for declarative validation annotations, and top-level docs are not synchronized.
- Are unit tests covering important paths and edge cases? Yes for the core implementation. Validation boundaries, mapping, sanitized error translation, and stubbed component flows are well covered. The main remaining risk is not unit coverage; it is repeatable end-to-end verification of live milestone creation.
- How is the current orchestration structure working? It is currently sound. `orchestration` owns the thin REST boundary, request mapping, validation, and use-case coordination; requests flow cleanly from controller to service to port to GitLab adapter and back. If the milestone capability grows further, the first cleanup should be splitting combined create/search validator responsibilities and promoting dates to typed orchestration values.

Final verdict: FAIL. The code is close to sign-off quality, but the current version should not pass review until the milestone-create smoke verification is made repeatable and green.
