## 2. Current Architecture and Request Flow

### Package Structure

```
orchestration/issues/
  IssuesService.java          — orchestrates port calls, composes EnrichedIssueDetail
  IssuesPort.java             — provider-agnostic port interface
  model/                      — orchestration domain models
    Change.java               — sealed interface for change payloads
    ChangeSet.java            — sealed interface for change-set entries
    ChangedBy.java            — actor record
    LabelChange.java          — implements Change (field=label)
    LabelChangeSet.java       — implements ChangeSet
    EnrichedIssueDetail.java  — composition of IssueDetail + changeSets
    IssueDetail.java          — single-issue detail (10 fields)
    Issue.java, IssuePage.java, IssueQuery.java, IssueState.java, CreateIssueInput.java
  rest/
    IssuesController.java     — thin REST endpoint
    dto/                      — IssueDetailDto (with inner ChangeSetDto, LabelChangeSetDto, etc.)
    mapper/                   — IssuesRequestMapper, IssuesResponseMapper

integration/gitlab/issues/
  GitLabIssuesAdapter.java    — implements IssuesPort via RestClient
  dto/                        — GitLab-specific response records
  mapper/                     — GitLabIssueDetailMapper, GitLabLabelEventMapper, GitLabIssuesMapper
```

### Request Flow for `GET /api/issues/{issueId}`

```
HTTP GET /api/issues/{issueId}
  → IssuesController.getIssueDetail()
    → IssuesService.getIssueDetail(issueId)
      → issuesPort.getIssueDetail(issueId)        ← SEQUENTIAL call 1
        → GitLabIssuesAdapter → GitLab API → GitLabIssueDetailMapper → IssueDetail
      → issuesPort.getLabelEvents(issueId)         ← SEQUENTIAL call 2
        → GitLabIssuesAdapter → GitLab API → GitLabLabelEventMapper → List<ChangeSet>
      → EnrichedIssueDetail(issueDetail, changeSets)   ← composition in orchestration ✓
    ← IssuesResponseMapper.toIssueDetailDto(enriched)
  ← IssueDetailDto JSON response
```

**Key observations about current flow:**
- Composition of multi-source data happens in `IssuesService` (orchestration layer) — correct per Constitution Principle 2.
- Integration adapters each map one external call to one orchestration contract — correct per Constitution Principle 2.
- The two port calls are **sequential**, not parallel — this is a performance concern.

## 4. Findings and Gaps

### 4.1 CRITICAL — Incomplete Orchestration Model: `Change` Interface Lacks Required Fields

**What is happening now:**

The `Change` interface in [Change.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/model/Change.java) defines only a single method:

```java
public interface Change {
    String field();
}
```

The `LabelChange` implementation adds `id` and `name` as record components, but these are not part of the `Change` contract. Any code working with `Change` at the interface level has no access to the common properties that every change type should carry.

Meanwhile, looking at the DTO layer in [IssueDetailDto.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/dto/IssueDetailDto.java#L44), `ChangeSetDto` is defined as an empty marker interface with zero contract:

```java
public interface ChangeSetDto {}
```

And `LabelChangeDto` defines `field`, `id`, and `name` — but `field` is not promoted to the `Change` interface contract in a useful way, and `id`/`name` are invisible at the `Change` level.

**Why this is a problem:**

1. **Unusable abstraction**: Any future code that works with `Change` polymorphically (e.g., domain logic evaluating change history) cannot access common change properties without downcasting. The interface is too thin to serve its purpose.
2. **Violation of model completeness**: The orchestration model should be the authoritative source of truth for what a "change" means in the domain. Right now `LabelChange` carries the real data, but the `Change` contract does not declare it.
3. **DTO divergence risk**: Since the domain model is incomplete, the DTO layer becomes the de-facto model definition — precisely the anti-pattern the user identified. If new change types are added, developers will copy the DTO shape rather than extending the domain model.

**What to prefer instead:**

The `Change` interface should declare the common fields that all change types must carry. At minimum:

```java
public interface Change {
    String field();     // enum-like type discriminator (currently "label", future: "status", "milestone", etc.)
    long id();          // ID of the changed resource
    String name();      // display name of the changed resource
}
```

This makes `Change` a usable contract. `LabelChange` already satisfies this — it just needs the interface to declare it. Future change types (`StatusChange`, `MilestoneChange`) will naturally implement the same contract.

**Expected benefit:** Domain logic can work with `Change` polymorphically. The model is self-documenting. New implementations are constrained by the contract, not by convention.

### 4.2 CRITICAL — DTO Contains Fields Not Present in Domain Model

**What is happening now:**

`LabelChangeDto` in [IssueDetailDto.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/dto/IssueDetailDto.java#L55) has three fields:

```java
public record LabelChangeDto(String field, long id, String name) {}
```

At the DTO level, `field` is a plain string. But at the domain model level, `Change.field()` returns a hardcoded `"label"` string from `LabelChange`:

```java
private static final String FIELD = "label";

@Override
public String field() {
    return FIELD;
}
```

There is no enum or typed discriminator at the domain level. The DTO carries a `field` property that looks like a type discriminator, but the domain model treats it as a magic string.

Additionally, `ChangeSetDto` is an empty marker interface — it provides no contract at all. Compare this with the domain `ChangeSet` interface which properly declares `changeType()`, `changedBy()`, `change()`, `changedAt()`. The DTO layer is weaker than the domain model in this regard.

**Why this is a problem:**

1. **Dirty and unmaintainable pattern**: The DTO layer invents its own structure rather than mirroring the domain 1:1. When domain logic evolves, the DTO shape and domain shape will drift apart.
2. **The `field` discriminator should be an enum**: `"label"` as a magic string is fragile. When `StatusChange` or `MilestoneChange` types are added, having no enum means no compile-time safety for the type discriminator.
3. **Empty `ChangeSetDto` interface**: Consumers of the API cannot rely on any common shape for change sets. JSON polymorphism will be needed but the contract gives zero guidance.

**What to prefer instead:**

1. Create a `ChangeField` enum in the orchestration model:
   ```java
   public enum ChangeField {
       LABEL
       // future: STATUS, MILESTONE, ASSIGNEE
   }
   ```
2. Update `Change` to use `ChangeField field()` instead of `String field()`.
3. `ChangeSetDto` should mirror `ChangeSet` with common accessor methods (`changeType`, `changedBy`, `change`, `changedAt`), not be an empty marker.
4. The response mapper should be a pure 1:1 translation from domain model to DTO — no structural invention.

**Expected benefit:** DTO mirrors domain model 1:1. Adding new change types requires extending the enum and implementing the interface — compile-time safety, no string-matching, no DTO-first development.

### 4.3 HIGH — Sequential Fetching in `getIssueDetail` Should Be Parallel

**What is happening now:**

In [IssuesService.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java#L64-L75):

```java
public EnrichedIssueDetail getIssueDetail(final long issueId) {
    log.info("Fetching issue detail issueId={}", issueId);
    final IssueDetail issueDetail = issuesPort.getIssueDetail(issueId);    // blocks
    final List<ChangeSet> changeSets = issuesPort.getLabelEvents(issueId);  // waits for above
    log.info("Label events fetched issueId={} count={}", issueId, changeSets.size());
    log.info("Issue detail fetched issueId={}", issueId);
    return EnrichedIssueDetail.builder()
            .issueDetail(issueDetail)
            .changeSets(changeSets)
            .build();
}
```

Both calls are to independent GitLab endpoints (`/issues/{id}` and `/issues/{id}/resource_label_events`). They share no data dependency — the label events call does not require the issue detail response. Yet they execute sequentially, meaning total latency = latency_A + latency_B.

**Why this is a problem:**

1. **Unnecessary latency**: Each call to GitLab involves network round-trip time. Sequential execution doubles the user-facing latency for this endpoint.
2. **Scalability concern**: As more data sources are composed (status events, milestone events, etc.), sequential calls will stack linearly. With 4 sources, the endpoint will be 4x slower than necessary.
3. **This is precisely the composition pattern where parallel execution adds clear value** without any complexity trade-off, because the calls are independent.

**What to prefer instead:**

Use `CompletableFuture` (or virtual threads with structured concurrency in Java 21+) to execute both calls in parallel:

```java
public EnrichedIssueDetail getIssueDetail(final long issueId) {
    log.info("Fetching issue detail issueId={}", issueId);

    final CompletableFuture<IssueDetail> detailFuture =
            CompletableFuture.supplyAsync(() -> issuesPort.getIssueDetail(issueId));
    final CompletableFuture<List<ChangeSet>> eventsFuture =
            CompletableFuture.supplyAsync(() -> issuesPort.getLabelEvents(issueId));

    final IssueDetail issueDetail = detailFuture.join();
    final List<ChangeSet> changeSets = eventsFuture.join();

    log.info("Issue detail and label events fetched issueId={} changeSetCount={}", issueId, changeSets.size());
    return EnrichedIssueDetail.builder()
            .issueDetail(issueDetail)
            .changeSets(changeSets)
            .build();
}
```

With Spring Boot 3.x on Java 21+, virtual threads make this even cleaner — each `supplyAsync` runs on a virtual thread with near-zero overhead.

**Important considerations:**
- Exception handling must be explicit: if one call fails, the other should be cancelled or its result discarded. `CompletableFuture.allOf()` or structured concurrency handles this.
- The `RestClient` used in the adapter must be thread-safe (it is — `RestClient` is thread-safe by design).
- Tests should verify that a failure in one port call propagates correctly even when the other succeeds.

**Expected benefit:** Endpoint latency is max(latency_A, latency_B) instead of sum. Scales cleanly as more data sources are composed.

### 4.4 MEDIUM — `ChangeSet` / `ChangeSetDto` Structural Symmetry is Broken

**What is happening now:**

The domain model has a well-structured hierarchy:
- `ChangeSet` interface → declares `changeType()`, `changedBy()`, `change()`, `changedAt()`
- `LabelChangeSet` record → implements `ChangeSet` with concrete types

But the DTO layer diverges:
- `ChangeSetDto` → **empty marker interface** with zero methods
- `LabelChangeSetDto` → has `changeType`, `changedBy`, `change`, `changedAt` — but the common interface declares none of these

This means:
1. API consumers cannot write generic code against `ChangeSetDto` — they must always downcast.
2. JSON serialization of polymorphic `ChangeSetDto` list will not include a type discriminator automatically — Jackson needs configuration or `@JsonTypeInfo` to handle this correctly.
3. The response mapper in [IssuesResponseMapper.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/mapper/IssuesResponseMapper.java#L85-L92) explicitly throws `IllegalArgumentException` for unsupported types, which is correct defensively — but the empty interface makes it invisible what shapes the API actually returns.

**Why this is a problem:**

The domain model got the abstraction right. The DTO layer should mirror it. An empty marker interface provides no contract and no compile-time safety for API consumers or for Jackson serialization.

**What to prefer instead:**

`ChangeSetDto` should be a sealed interface (or at minimum declare common accessors) that mirrors `ChangeSet`:

```java
public sealed interface ChangeSetDto permits LabelChangeSetDto {
    String changeType();
    ChangedByDto changedBy();
    OffsetDateTime changedAt();
}
```

This gives API consumers a usable contract and makes Jackson serialization predictable.

**Expected benefit:** DTO layer mirrors domain model. API consumers can work with change sets generically. Jackson can serialize polymorphic types correctly.

### 4.5 MEDIUM — Logging Gaps in Orchestration Layer

**What is happening now:**

Logging is generally good at the adapter level (each operation logs entry, exit, and failure with contextual fields). However, there are gaps in the orchestration layer:

1. **`IssuesService.getIssueDetail()`** logs "Label events fetched" and "Issue detail fetched" as two separate log lines after both calls complete. The "Issue detail fetched" log comes *after* the label events log — the order is misleading since the detail was fetched first. More importantly, there is no log line recording the total composition time or the combined result shape.

2. **No timing information**: For an endpoint that composes multiple external calls, timing is critical for operational monitoring. Neither the individual call durations nor the total composition duration is logged.

3. **`IssuesService.getIssues()`** logs filter presence as booleans (`state:true`, `labels:true`) rather than the actual filter values. This makes debugging harder — knowing *which* state or label was requested matters for diagnosing unexpected results.

**Evidence:**

[IssuesService.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java#L30-L38) — filters logged as booleans:
```java
log.info("Requesting issues page={} perPage={} filters=[state:{},labels:{},assignee:{},milestone:{}]",
    query.page(), query.perPage(),
    query.state() != null, query.label() != null,
    query.assignee() != null, query.milestone() != null);
```

[IssuesService.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/IssuesService.java#L68-L70) — misleading log order:
```java
final List<ChangeSet> changeSets = issuesPort.getLabelEvents(issueId);
log.info("Label events fetched issueId={} count={}", issueId, changeSets.size());
log.info("Issue detail fetched issueId={}", issueId);  // logged after label events
```

**What to prefer instead:**

1. Log filter values directly (they are not sensitive — they are user-provided search criteria): `state={}, label={}, assignee={}, milestone={}`.
2. Merge the two completion logs into one: `log.info("Issue detail composed issueId={} changeSetCount={}", issueId, changeSets.size())`.
3. Consider adding duration logging for the composition operation (especially once parallel execution is implemented).

**Expected benefit:** Logs are accurate, unambiguous, and operationally useful. Debugging filter-related issues becomes possible without code inspection.

### 4.6 LOW — Mapper Structural Duplication Between Response Mapper Layers

**What is happening now:**

[IssuesResponseMapper.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/rest/mapper/IssuesResponseMapper.java#L49-L80) and [GitLabIssueDetailMapper.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/integration/gitlab/issues/mapper/GitLabIssueDetailMapper.java) both map assignee lists and milestone objects with nearly identical patterns:

**Response mapper** (orchestration → DTO):
```java
final var assignees = issueDetail.assignees().stream()
    .map(a -> AssigneeDto.builder().id(a.id()).username(a.username()).name(a.name()).build())
    .toList();
```

**Detail mapper** (GitLab → orchestration):
```java
final List<AssigneeDetail> assignees = response.assignees().stream()
    .map(a -> AssigneeDetail.builder().id(a.id()).username(a.username()).name(a.name()).build())
    .toList();
```

The same structural translation repeats for milestone. This is expected from explicit mapping at every boundary (Constitution Principle 8) — each mapper owns its boundary. The observation is merely about awareness: as more sub-models are added, this duplication will grow proportionally.

**Why this is noted (not a problem yet):** This is the cost of honest boundaries. It only becomes a problem if the shapes diverge and someone copies from the wrong mapper. No action required now, but worth monitoring.

### 4.7 LOW — Missing Edge-Case Tests for Label Event Mapping

**What is happening now:**

[GitLabLabelEventMapperTest.java](../../flow-orchestrator/src/test/java/com/gitlabflow/floworchestrator/integration/gitlab/issues/mapper/GitLabLabelEventMapperTest.java) covers:
- Happy path (single event mapping)
- Empty list input
- Missing label field

**Missing edge cases:**

1. **Missing `user` field**: The mapper calls `requiredField(response.user(), "user")` — but there is no test for `user == null`. Only `label == null` is tested.
2. **Missing `action` field**: The mapper calls `requiredField(response.action(), "action")` — no test for `action == null`.
3. **Missing `createdAt` field**: The mapper calls `requiredField(response.createdAt(), "created_at")` — no test for `createdAt == null`.
4. **Multiple events**: Only single-event mapping is tested. A test with 2+ events would verify list ordering is preserved.

Similarly, [GitLabIssueDetailMapperTest.java](../../flow-orchestrator/src/test/java/com/gitlabflow/floworchestrator/integration/gitlab/issues/mapper/GitLabIssueDetailMapperTest.java) covers nullable fields well but does not test:
- Multiple assignees (only single-assignee or empty lists)
- Assignee field-level nullability (what happens if `username` or `name` is null in the GitLab response)

**Why this matters:** These are boundary validation checks that protect against upstream API changes. If GitLab changes their response shape, these tests would catch regressions.

**Expected benefit:** Complete required-field coverage gives confidence that the mapper fails clearly on malformed upstream data.

### 4.8 OBSERVATION — `IssueState` Enum Uses String Values Instead of Direct Serialization

**What is happening now:**

[IssueState.java](../../flow-orchestrator/src/main/java/com/gitlabflow/floworchestrator/orchestration/issues/model/IssueState.java) has a manual `value` field and `fromValue()` method:

```java
public enum IssueState {
    OPENED("opened"), CLOSED("closed"), ALL("all");
    // ...
    public static IssueState fromValue(final String value) {
        return switch (value.toLowerCase(Locale.ROOT)) { ... };
    }
}
```

This is functional and perfectly correct. The observation is that `@JsonValue`/`@JsonCreator` annotations would make this work automatically with Jackson serialization, but that would introduce a framework dependency in the orchestration model — which Constitution Principle 3 discourages (domain/orchestration models should not carry serialization annotations). The current manual approach is actually the right choice for this project.

No action needed. This is confirmation that the current pattern is correct.

## 6. Final Verdict

The codebase is **structurally sound** at the architecture level. The v2.3.0 refactoring successfully established correct layer boundaries — composition happens in orchestration, integration adapters map one call to one contract, defensive copies are consistent, error handling is centralized, and the test infrastructure is well-organized across three levels plus Karate smoke tests.

**The primary architectural debt is in the modeling layer.** The `Change`/`ChangeSet` hierarchy has the right shape (interfaces + sealed implementations) but the contracts are incomplete. The `Change` interface is too thin to be useful, the `field` discriminator is a magic string instead of an enum, and the DTO layer introduces structural divergence (empty `ChangeSetDto` marker) rather than mirroring the domain 1:1.

**The secondary concern is performance.** Sequential fetching of independent data sources is a known anti-pattern for composition endpoints. With only two sources this is tolerable but not acceptable as a pattern for growth.

**Recommendation:** Address P0 (model enrichment) and P1 (parallel fetching + logging) before adding the next capability. The current modeling gaps will compound with every new change type (status events, milestone events, etc.) and become significantly harder to fix retroactively.

The codebase is safe to keep as the working baseline, but the modeling refactoring should be the first action item in the next iteration — before any new feature work.
